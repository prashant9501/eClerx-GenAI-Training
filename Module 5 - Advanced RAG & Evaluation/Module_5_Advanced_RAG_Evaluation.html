Part 1

Why basic RAG
plateaus

Module 4 gave you a working pipeline. Today is about why that pipeline tops out at ~60% accuracy on hard questions — and the four levers you pull to push past that.

The four failure modes of basic RAG

① Embedding ≠ keyword overlap

Dense vectors miss exact-match queries — codes, IDs, ticker symbols, regulation numbers. Pure semantic retrievers lose to BM25 on these.

② Single query, single perspective

One user query → one embedding → one set of neighbours. Misses chunks that paraphrase the same idea differently.

③ Chunks lose their context

"The cap is 5%" — 5% of what? The chunk knew, but you split it away from "single-stock concentration".

④ Top-K is unranked by relevance

Vector similarity ≠ relevance to the question. A re-ranker that scores the (query, chunk) pair consistently beats raw vector order.

Each failure mode maps to a specific technique in Part 2: hybrid (① ), multi-query / HyDE (② ), contextual retrieval (③ ), cross-encoder rerank (④ ).

Part 2

Advanced
retrieval

Multi-query, HyDE, hybrid (BM25 + semantic), MMR diversity, and cross-encoder reranking — the Lab 5.1 toolkit.

Lever 1 — Multi-Query & HyDE

MultiQueryRetriever

LLM rewrites the user's query into N paraphrases. Each retrieves independently; results are union-ed and de-duped.

retriever = MultiQueryRetriever.from_llm(
    retriever=base, llm=llm,
)

Fixes failure mode ② — single query, single perspective.

HyDE — Hypothetical Document Embeddings

Generate a plausible answer with the LLM first; embed that and search. Why? Question embeddings live in question-shaped space; answer embeddings live in document-shaped space.

Fails when the LLM hallucinates a wrong answer that pulls bad context.

Part 2 · 1/4

Lever 2 — Hybrid: BM25 + Semantic via RRF

BM25 wins on exact-match (codes, IDs, regulation numbers); dense vectors win on paraphrase. Combine them with Reciprocal Rank Fusion:

RRF(d) = Σ_retrievers 1 / (k + rank_r(d))
# k=60 is standard
hybrid = EnsembleRetriever(
    retrievers=[bm25, dense],
    weights=[0.4, 0.6],
)
  • RRF doesn't need calibrated scores across retrievers — it ranks ranks.
  • The weights parameter biases the merge.
  • Tune weights per dataset; defaults rarely optimal.
Part 2 · 2/4

Lever 3 — MMR diversity

Top-K by raw similarity often returns N near-duplicate chunks of the same text. MMR (Maximal Marginal Relevance) re-ranks to balance relevance against novelty:

MMR(d) = λ · sim(d, q)
       − (1 − λ) · max_d' sim(d, d')
# λ = 1.0 → pure relevance
# λ = 0.0 → pure diversity
# λ = 0.5 → balanced (default)
  • Reach for it when chunks are short and redundant.
  • Skip it when each retrieved chunk is already distinct.
  • Tune lambda_mult on a labelled dev set, not a feeling.

When MMR is a no-op

If your top-5 are already from 5 different documents, MMR can't improve diversity any further. Inspect a few queries before turning it on globally.

When MMR hurts

If the question genuinely needs multiple supporting chunks from the same doc (a long legal clause), MMR will penalise the second one as redundant — and you lose recall.

Part 2 · 3/4

Lever 4 — Cross-Encoder Reranking

Embeddings are bi-encoder: query and chunk are encoded independently, scored by cosine. Cheap, but imprecise — relevance is not just geometric similarity.

A cross-encoder takes the (query, chunk) pair as one sequence and scores them jointly. ~100× slower per pair, but dramatically more accurate.

Two-stage retrieval: cheap bi-encoder for top-50 → expensive cross-encoder for top-5.

reranker = CrossEncoderReranker(
    model=ms_marco_minilm, top_n=5
)
ContextualCompressionRetriever(
    base_compressor=reranker,
    base_retriever=hybrid,
)

Part 3

Contextual
retrieval

Anthropic's 2024 trick: tell every chunk where it came from. Lab 5.2 measures the gain — and the cost.

The chunk-context problem

Naive chunk

"The cap is 5%. Single-issuer
exposure must not exceed 2%
in standard accounts."

Embedding has no idea this is about fixed-income high-yield exposure. A vector search for "high-yield concentration limit" misses it.

Contextual chunk

"From IPS-FI-001 (Fixed Income
Policy, High-Yield section): The
cap is 5%. Single-issuer exposure
must not exceed 2% in standard
accounts."

Now the embedding lives in a much more discriminating region — and BM25 picks up the new tokens too.

Mechanic. Before embedding, prepend a 50–100 token preamble generated by Claude that locates the chunk in its parent document. The chunk's surface text is unchanged in the LLM's context window — only the embedding changes.
Part 3 · 1/2

The cost / benefit calculus

Cost

  • One Claude call per chunk at index time.
  • ~$1–3 per million tokens of corpus (with prompt caching, much less).
  • Re-runs only when documents change.

Benefit (Anthropic 2024 paper)

  • ~35% retrieval-failure reduction with contextual embeddings.
  • ~49% with BM25 + contextual.
  • ~67% with reranker on top.

When it's worth it

  • Long documents with section structure (legal, medical, financial).
  • Chunks that lose meaning in isolation.
  • High-precision retrieval requirements (compliance, finance).

When it isn't

  • Self-contained chunks (FAQ, Q&A pairs).
  • Tiny corpora — pay the eval cost first, then decide.
  • Workloads dominated by exact-match queries (BM25 handles those).

Part 4

Citations
& agentic RAG

Surface where the answer came from. And let the model retrieve more when one pass isn't enough.

Source citations — the audit surface

For finance, legal, healthcare — every answer needs a clickable citation back to a source paragraph. Two patterns:

① Retrieval-side citations

Surface the doc_id + chunk_id + page number of every retrieved chunk in the response payload. Easy. Doesn't tell you which chunk supported which claim.

② Generation-side citations

Prompt the LLM to inline-cite each claim: "VaR limit is 2% [IPS-RISK-001]". Then post-validate that every cited doc_id is actually in the retrieved set — never hallucinate a citation.

Hard rule for regulated workloads

Every fact in the response must trace to a chunk in the retrieved set. Validate it programmatically — don't trust the LLM to follow the prompt.

def validate_citations(answer, retrieved):
    cited = re.findall(r"\[([A-Z0-9-]+)\]", answer)
    available = {d.metadata["doc_id"]
                 for d in retrieved}
    bogus = set(cited) - available
    if bogus:
        raise CitationError(bogus)
Part 4 · 1/2

Agentic RAG — when one pass isn't enough

Single-pass RAG: retrieve once → answer. Fine for direct questions; brittle when the user's question requires lookups the model only realises after seeing the first result.

Agentic RAG wraps retrieval as a tool the LLM calls in a loop:

  • Retrieve, read, decide whether to retrieve more.
  • Decompose multi-hop questions ("What's the policy on X, and how does it interact with Y?").
  • Refine the query based on what was found (or not).

Cost: 2–5× the LLM calls of single-pass RAG; latency penalty too. Use only when the question class actually benefits — not by default.

@tool
def retrieve_policy(query: str) -> str:
    """Search Meridian policies."""
    docs = retriever.invoke(query)
    return format_with_citations(docs)

agent = create_agent(
    llm,
    tools=[retrieve_policy],
    prompt=AGENTIC_RAG_PROMPT,
)
agent.invoke({"messages": [...]})
Lab 5.2 builds this end-to-end and benchmarks single-pass vs agentic on the same 20-question eval — measurable accuracy gain, but observe the latency / cost cost.

Part 5 · 1/2

When to reach for LlamaIndex

LlamaIndex wins

  • Multiple heterogeneous indices needing routing — RouterQueryEngine is built-in.
  • Multi-hop / decomposed questions — SubQuestionQueryEngine is built-in.
  • RAG is the centre of gravity of the app.
  • You want first-class summary / keyword / knowledge-graph indices alongside vector.

LangChain wins

  • App is conversational + tool-heavy with RAG as one sub-system.
  • You want LangSmith / LangFuse first-class observability.
  • Team already deep in LangChain — cognitive cost of a second framework matters.
  • You need rich retriever ensembles (BM25 + dense + rerank) on the same store.
Practical rule. Most production stacks use both — LlamaIndex for the RAG sub-system, behind a LangChain (or FastAPI) orchestration layer. Lab 5.3 walks the LlamaIndex path end-to-end so you can defend either choice.

Part 6

RAG
evaluation

RAGAS for end-to-end quality, deterministic IR metrics for retrieval-only ablations, and the LLM-as-Judge biases you have to design around. Lab 5.4.

Two halves of the harness

① Retrieval-only metrics (deterministic)

  • Hit Rate@K — did any relevant doc appear in top-K?
  • MRR — 1/rank of first relevant doc.
  • Precision@K — fraction of top-K that's relevant.
  • Recall@K — fraction of relevant docs we caught.
  • nDCG@K — graded ranking quality.

Cheap. Reproducible. Run in CI.

② End-to-end quality (RAGAS · LLM judge)

  • Faithfulness — every claim follows from context.
  • Answer Relevancy — answer addresses the question.
  • Context Precision — top-ranked chunks are the most relevant.
  • Context Recall — retrieval brought back everything needed.

Expensive. Stochastic. Catches what IR metrics miss.

You need both. IR metrics tell you whether retrieval got the right chunks; RAGAS tells you whether generation used them faithfully. A regression on one isolates which half broke.
Part 6 · 1/3

RAGAS coverage diagram

Faithfulness vs Relevancy can disagree

Faithful but not relevant: "The provided context discusses fixed-income duration." — every claim follows from context, but the user asked about equities.

Relevant but not faithful: "Maximum equity allocation is 95%" when policy says 90%. Addresses the question but invents the number.

That's why both metrics ship in RAGAS — neither alone is sufficient.

Part 6 · 2/3

The LLM-as-Judge biases you have to design around

BiasWhat happensMitigation
PositionJudge prefers position A or B regardless of quality.Randomise; run both orders; average.
VerbosityLonger answers preferred, often inaccurately.Length-normalise scores; cap answer length.
Self-enhancementJudge prefers answers that look like its own style.Use a different model family as judge.
SycophancyJudge agrees with the framing of the question.Neutral, structured rubrics; never frame "right" answer in the prompt.
Family-resemblanceSame model family scores higher.Cross-family judge ensembles.
Prompt-leakVisible reasoning biases the judge upward.Strip CoT; judge only the final answer.
Why it matters. RAGAS and any custom LLM judge inherit all of these. Combining them with deterministic IR metrics (the left card on the previous slide) gives you a tripwire: if RAGAS drifts but IR metrics don't, the judge is the problem, not the system.

Three checkpoints for Day 5

① Retrieval levers

For any retrieval failure mode, can you point to the lever that fixes it? (BM25 / hybrid → exact-match miss; multi-query → query phrasing; rerank → ranking quality; contextual → chunk isolation.)

Goal: stop guessing; stack levers deliberately.

② Framework fluency

Articulate in one breath: when to reach for LlamaIndex, when to stay in LangChain, when to use both. Defend either choice in front of an interviewer.

Goal: framework choice on the merits, not on familiarity.

③ Evaluation discipline

Every retrieval/synthesis change is justified by a number, not a vibe. RAGAS for generation; Hit@K / MRR / nDCG for retrieval. Sweep K. Compare variants on the same dataset.

Goal: "we improved retrieval by 12 points on nDCG@4" — not "it feels better".
