Part 1

Why observability
for LLM apps

Stochastic outputs · multi-step chains · third-party endpoints · per-call dollar costs · hallucinations that look correct. The traditional logging stack was not designed for this.

The new operational reality

  • Non-determinism. Same input, different output — even at temperature 0.
  • Token cost is per call. Every retry, every sub-chain, every tool call shows up on the bill.
  • Latency is variable. P50 ≠ P95; tail latencies blow SLAs.
  • Hallucinations look correct. A confident, fluent answer hides the wrong order ID.
  • Multi-step chains. Failures rarely happen at the surface — they're 4 nodes in.
  • Third-party endpoints. You can't strace a model running in someone else's data centre.
What you need to see. Per-request: prompt, tool calls (name + args), per-step latency, token in/out, model + version, error trace, judge score, user-id, session-id, deployment env.
What you need to query. "Show me all fraud-router queries last hour with quality score < 3 and latency > 5 s." That's not grep. That's a tracing platform.
What you need to gate. A new prompt or model version should not ship until it scores ≥ baseline on the eval suite. Eval = release gate, not retro dashboard.
Part 1 · 1/2

The four pillars of LLM-app observability

All four pillars compose. A trace platform feeds an eval pipeline; eval scores show up in audit logs; audit logs trigger fairness guardrails. The order matters: traces first — without them, you can't debug, evaluate, or audit anything.

Part 2

LangSmith

The first-party tracing + eval platform from the LangChain team. Three env-vars and every .invoke in your code starts producing structured traces.

Setup is three environment variables

export LANGSMITH_TRACING=true
export LANGSMITH_API_KEY=ls_...
export LANGSMITH_PROJECT="customer-support-router"

No code changes. Every LangChain / LangGraph .invoke, .stream, .batch emits a trace tree to LangSmith Cloud (or your self-hosted instance). Tool calls, retries, parser failures, model + version, token counts — all captured automatically.

Auto

chain · sub-chain · LLM call · tool call · output parser · retries

Manual

add custom metadata / tags via RunnableConfig

Filterable

by user-id · session · model · latency · error · tag

Datasets · Evaluators · Experiments

ConceptWhat it isWhen to add
DatasetA versioned list of {input, expected_output} examples.Day-1 — even 20 hand-curated cases beats nothing.
EvaluatorA function (deterministic, LLM-as-Judge, or human) that scores one example.One per quality dimension you care about.
ExperimentRuns a chain over a dataset, applies evaluators, stores the matrix of results.Every prompt or model change → new experiment → diff.
 
from langsmith import Client; from langsmith.evaluation import evaluate
client = Client()
client.create_dataset("support-router", description="20 routing test cases")
def correctness(run, example): return {"score": run.outputs["category"] == example.outputs["category"]}
results = evaluate(my_chain.invoke, data="support-router", evaluators=[correctness])
Part 2 · 2/3

LangSmith Hub for prompt versioning

Why. A prompt is a piece of source code. Versioning it in the Hub lets every commit be reviewed, rolled back, A/B-tested.
Pattern. hub.push("my-team/router", prompt) from a CI job; hub.pull("my-team/router") at chain init time. Lock to a SHA in production.
from langchain import hub
# CI: publish on prompt change
hub.push("eclerx/support-router-v3", prompt_template,
         parent_commit_hash="abc123")

# Runtime: pull a specific version
prompt = hub.pull("eclerx/support-router-v3:b7d4e9f")
chain = prompt | llm | StrOutputParser()
"Prompts that ship to production are source code. Treat them like source code — review, version, rollback, test."

Part 3

Custom callbacks
for audit trails

When LangSmith Cloud isn't allowed (data residency) or when you need a CloudWatch-shaped, SIEM-friendly JSON-lines record per request.

The callback hook timeline

Subclass BaseCallbackHandler, override the hooks you need, append a JSON-lines record per event to your audit sink. Compose with retries, latency timers, and PII-redactors — all observe the same chain without modifying it.

Part 3 · 1/2

Audit-handler skeleton

from langchain_core.callbacks import BaseCallbackHandler
import time, json, uuid

class AuditHandler(BaseCallbackHandler):
    def __init__(self, sink_path: str): self.sink_path, self._t = sink_path, {}
    def on_llm_start(self, serialized, prompts, run_id, **kw):
        self._t[run_id] = time.time()
    def on_llm_end(self, response, run_id, **kw):
        usage = response.llm_output.get("token_usage", {}) if response.llm_output else {}
        self._write({"event":"llm_end","run_id":str(run_id),"latency_ms":int((time.time()-self._t[run_id])*1000),
                     "tokens_in":usage.get("input_tokens"), "tokens_out":usage.get("output_tokens")})
    def on_tool_end(self, output, run_id, **kw):
        self._write({"event":"tool_end","run_id":str(run_id),"output_preview":str(output)[:120]})
    def _write(self, rec): open(self.sink_path,"a").write(json.dumps(rec)+"\n")

agent.invoke({"messages":[...]}, config={"callbacks":[AuditHandler("audit.jsonl")]})
Regulatory pattern. JSON-lines + PII-redacted + write-once + 7-year retention satisfies most banking record-keeping rules. Stream to S3 → Glacier for long-term tier.

Three integration patterns · production decision

Decorator API

from langfuse.decorators import observe

@observe()
def run_pipeline(q):
    return chain.invoke(q)

Best for plain Python functions outside LangChain.

LangChain callback

from langfuse.langchain import CallbackHandler

cb = CallbackHandler(user_id="...")
chain.invoke(q, config={"callbacks":[cb]})

Same pattern as LangSmith — drop-in.

OpenAI SDK wrap

from langfuse.openai import openai

# every openai.chat.completions.create()
# is auto-traced

For raw OpenAI / Anthropic SDK code.

Decision rule. Default to LangSmith for dev velocity and the prompts hub. Use Langfuse self-hosted when (a) regulator forbids cloud trace platforms, (b) you need data residency in your VPC, or (c) license cost crosses a few hundred users.

Routing accuracy · response quality · trajectory correctness. Each catches a failure mode no other layer can. Plus pass@k for stochastic brittleness.

The 3-layer eval hierarchy

"A response can score 5/5 on quality, route correctly, AND cite the wrong customer's order. Without L3, you'd never know."
Part 5 · 1/2

pass@k · LLM-as-Judge calibration

pass@k · Run each test k times. pass@1 = single-try accuracy; pass@k = at least one of k tries passes. The gap quantifies stochastic brittleness — useful even at temperature 0.
for trial in range(K):
    out = agent.invoke(test["query"])
    trials.append(grade(out))
pass_at_1 += int(trials[0])
pass_at_k += int(any(trials))
LLM-as-Judge biases (and fixes).
  • Self-preference. Same-family judges inflate ~3–7%. Use a different family (OpenAI judging Anthropic, etc.).
  • Position bias. A vs B comparisons favour the first option. Randomise order.
  • Verbosity bias. Judges over-reward longer answers. Anchor the rubric to specific criteria, not "thoroughness".
  • Calibration. Hold a small human-rated gold set; track judge-vs-human Cohen's κ over time.

Part 6

Skills as prompts +
prompt-optimization loop

A skill = a self-contained instruction pack. Load on demand, A/B test, optimize the weakest, ship measurable improvements.

Skills-as-prompts pattern

One giant prompt

System message tries to cover claims + support + compliance in one go. Token cost high. Updates risky. Model often mixes domains.

Skills registry

One dict per skill: {name, system_prompt, rules, few_shot}. Classifier picks one. Only that skill is loaded. Updates isolated. Cheap tokens.

SKILLS = {"claim_processing": {...}, "customer_support": {...}, "compliance_review": {...}}
def run_skill_agent(q):
    key = classify_skill(q)
    sys = SKILLS[key]["system_prompt"] + "\n" + SKILLS[key]["rules"]
    return (ChatPromptTemplate.from_messages([("system",sys),("user","{q}")]) | llm).invoke({"q":q})
Two patterns. (a) Tool-loaded: the agent calls load_skill as a tool — extra round-trip but fully agentic. (b) Dynamic system prompt (above): classify-then-load — cheaper and faster. Default to (b) for production.

The eval-driven optimization loop

Every prompt change is justified by a measurable score delta on a fixed test set. No vibes-based tuning. This is the engineering loop that separates a research demo from a production product.

PII redaction pipeline as a callback

def redact_pii(text: str) -> tuple[str, list[dict]]:
    pii = comprehend.detect_pii_entities(Text=text, LanguageCode="en")["Entities"]
    redacted = text
    # Sort right-to-left so each splice doesn't shift the next offset
    for p in sorted(pii, key=lambda e: e["BeginOffset"], reverse=True):
        redacted = redacted[:p["BeginOffset"]] + f"[{p['Type']}]" + redacted[p["EndOffset"]:]
    return redacted, pii

# Compose into the chain BEFORE the model
chain = RunnableLambda(lambda s: {**s, "user_text": redact_pii(s["user_text"])[0]}) \
        | summarise_prompt | llm

Comprehend wins

cheap · fast · deterministic · span-accurate · regulator-friendly audit story

Comprehend gaps

Aadhaar / PAN / GSTIN · context spread across sentences · custom internal IDs

Production pattern

Comprehend first → regex pass for India IDs → LLM fallback on high-stakes channels

Part 7 · 1/3

Fair-Lending pipeline with the Four-Fifths guardrail

def fairness_guardrail(decisions, attrs, threshold=0.80):
    for a in attrs:
        rates = decisions.groupby(a)["approved"].mean()
        if (rates / rates.max() < threshold).any():
            raise FairnessGuardrailFailure(f"{a}: ratio < 0.80 — release blocked")
Part 7 · 2/3

Disparate impact vs disparate treatment

Disparate treatment (intent)

The model uses a protected attribute (race / gender / religion) as an explicit input. Illegal regardless of outcome.

Easy to defend: just don't pass the attribute.

Disparate impact (outcome)

The model's decisions correlate with a protected attribute even though it isn't an input. Tested via the Four-Fifths Rule.

Hard to defend: proxies (zip code, name, credit-score history) leak the attribute even when removed from inputs. Requires business-necessity justification.

Lab 9.4 finding. The agent never received race or gender — and still produced disparate impact via the income / credit-score proxies. That is the textbook fair-lending failure mode. Fairness audits must measure outcomes, not just check that the protected attribute isn't passed in.
