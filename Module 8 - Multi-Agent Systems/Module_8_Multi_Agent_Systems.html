Part 1

Why multi-agent —
when single agent + tools
is not enough

Module 7 ended with a tool-calling agent that could research, draft, and review on its own. So why split that into a team? The answer is the seam between specialisation, scale, and auditability.

Three forces that push you to multi-agent

1. Context-window bloat

A single agent that knows all 27 banking tools, all 14 retrieval policies, and all 8 compliance rules ends up with a 12k-token system prompt. It's slow, expensive, and confused. Split into specialists with 3-tool prompts each.

2. Role-specific tone & backstory

The Researcher should be neutral and citation-heavy. The Writer should be marketing-tonal. The Compliance Reviewer should be paranoid. One system prompt can't be all three.

3. Audit + HITL boundaries

Regulators want to see: "who decided this?". An explicit specialist node is auditable. A monolithic agent's chain-of-thought isn't. HITL interrupts also need clear boundaries.

Heuristic: if your single-agent prompt is > 2 000 tokens, your tool list > 8, or you find yourself adding "if context says X then behave like Y" rules — you're past the single-agent threshold.
Part 1 · 1/2

Three coordination patterns we'll see today

Supervisor

One LLM router decides which specialist runs next. Specialists return to the supervisor; supervisor decides "done" or "next". Best for triage / customer-service.

Lab 8.1 · LangGraph

Orchestrator-Worker

One LLM plans an unknown number of subtasks; workers run in parallel via the Send API; an evaluator decides accept-or-revise. Best for variable-shape work (reports, batches).

Lab 8.2 · LangGraph

Evaluator-Optimizer

An LLM judge scores the worker's output and decides revise-or-ship. Bounded loop with a max-iteration safeguard. Often layered on top of orchestrator-worker.

Lab 8.2 · LangGraph

These three are patterns, not frameworks. You can implement all three in LangGraph; CrewAI ships sequential / hierarchical equivalents; MAF ships RoundRobinGroupChat / SelectorGroupChat; OpenAI SDK ships handoffs.

Part 2

Supervisor pattern
Command(goto=...)

One router LLM, many specialists, a shared TypedDict state. The bread-and-butter pattern of customer-service multi-agent — and the basis of Lab 8.1's ShopSmart support build.

The supervisor in one diagram

Three things to notice

  • The supervisor is itself an LLM call — it picks the next node by name.
  • Routing happens via Command(goto="loan_agent") — explicit, traceable.
  • Specialists return to the supervisor; only the supervisor decides "done".

Why this beats a chain of if/elif

A regex/keyword router fails on ambiguous queries ("I want to check my home-loan EMI and also transfer some money"). An LLM router handles ambiguity, multi-intent, and vague phrasing — at the cost of one extra LLM call per turn.

Lab 8.1 §6 walks through the supervisor's prompt — note how it lists each specialist's capabilities, not just its name.
Part 2 · 1/2

Command(goto=...) — the handoff primitive

from langgraph.types import Command

def supervisor(state: State) -> Command:
    decision = llm.invoke(
        f"Route the query to one of: "
        f"account, loan, investment, end. "
        f"Query: {state['messages'][-1].content}"
    )
    next_node = parse(decision.content)
    if next_node == "end":
        return Command(goto=END)
    return Command(
        goto=next_node,
        update={"hops": state["hops"] + 1},
    )

Two things in one return: where to go next and what to update in shared state.

Three guardrails Lab 8.1 adds

  • Hop counter in state — supervisor stops after state["hops"] >= 6 to prevent loops.
  • Explicit END node — supervisor must decide the conversation is done; no implicit fall-through.
  • Telemetry — log every routing decision with reasoning to a callback handler (Module 9 picks this up).
Without a hop counter, two specialists can ping-pong forever ("defer to loan" → "no, defer to account" → ...). It's the most common multi-agent bug.

+ Evaluator-Optimizer

When the work isn't a triage but a plan-fan-out-evaluate loop — Lab 8.2's compliance-report generator with HITL approval mid-graph.

Orchestrator-Worker — fan-out via Send

Why Send, not a fixed branch

The orchestrator doesn't know how many sections the report will have until it sees the brief. A 2-section brief should not pay for 5 worker nodes. Send spawns one worker per planned subtask at runtime.

from langgraph.types import Send

def fan_out(state):
    return [
        Send("worker", {"section": s})
        for s in state["plan"].sections
    ]

Workers run in parallel

For 5 sections @ ~8s each: serial = 40s, parallel via Send = ~9s. Same wall-clock pattern as RunnableParallel from Module 3.

Part 3 · 1/2

Evaluator-Optimizer + HITL

Evaluator-as-LLM

The evaluator is a small dedicated LLM agent whose only job is to return a structured Pydantic verdict — {verdict: APPROVE | REVISE, reasons: [...]}. Bounded loop with max_iters = 3.

def evaluator(state):
    v = judge_llm.invoke(state["draft"])
    if v.verdict == "APPROVE" or state["iters"] >= 3:
        return Command(goto="hitl")
    return Command(
        goto="reviser",
        update={"iters": state["iters"]+1,
                "feedback": v.reasons},
    )

HITL via interrupt()

Mid-graph the workflow pauses; a human compliance officer reviews and either approves or annotates. Command(resume=approval) continues.

from langgraph.types import interrupt

def hitl(state):
    approval = interrupt({
        "draft": state["draft"],
        "evaluator": state["feedback"],
    })
    return {"approval": approval}

Microsoft Agent Framework
autogen-agentchat 0.4 lineage

Lab 8.3 — the unified evolution of AutoGen 0.2 and Semantic Kernel, with Azure AI Foundry on top. The open-source core is autogen-agentchat ≥ 0.4.

The lineage — and why the package name matters

The three checkpoints

  • 2023 — pyautogen (AutoGen 0.2). Synchronous, single-threaded, hard to scale. Now legacy.
  • 2024 — autogen-agentchat ≥ 0.4 + autogen-ext. Async core, group chat patterns, model-provider extensions. This is what you pip-install.
  • 2025 — Microsoft Agent Framework (MAF). Same open-source core, plus Azure AI Foundry integration (managed runtime, identity, monitoring).
Do not install pyautogen. Lab 8.3 explicitly pins autogen-agentchat >= 0.4. Mixing the two corrupts your env.
Part 4 · 1/2

The four MAF building blocks Lab 8.3 uses

AssistantAgent

The base agent. Wraps an LLM + system message + (optional) tools. Async-first.

RoundRobinGroupChat

Cycles through a list of agents in fixed order. Equivalent to CrewAI's sequential.

SelectorGroupChat

An LLM picks who speaks next from the group. Equivalent to LangGraph's supervisor pattern.

Handoff

Explicit agent-to-agent transfer. Like OpenAI SDK's handoffs but inside MAF's group-chat abstraction.

When does MAF beat LangGraph for a financial-services client? When the client is already on the Azure stack — Foundry adds managed identity, regional data residency, and built-in monitoring that you'd otherwise wire up by hand. For multi-cloud or AWS-native clients, LangGraph stays ahead.

CrewAI
role · goal · backstory

Lab 8.4 — when the workflow already maps onto an org chart, this is the framework that gets you to a demo fastest.

The CrewAI mental model

An agent is a team member

researcher = Agent(
    role="Fintech Market Research Analyst",
    goal="Find the latest market trends "
         "and data points for content briefs",
    backstory=(
        "8 years tracking Indian fintech. "
        "Follows NPCI, SEBI, RBI daily. "
        "Knows UPI, MFs, digital lending."
    ),
    tools=[search_market_trends],
    llm="openai/gpt-4o-mini",
)

The backstory acts as a permanent, role-scoped system prompt that biases tone, vocabulary, and risk appetite before any task arrives.

A task is what they do

writing_task = Task(
    description=(
        f"Write a {brief['platform']} post "
        f"about {brief['topic']}."
    ),
    agent=writer,
    expected_output=(
        f"A complete {brief['platform']} post "
        f"ready for editorial review."
    ),
)
expected_output is more important than it looks — it's the contract the next task in the chain assumes.
Part 5 · 1/2

Two processes — sequential vs hierarchical

Process.sequential

Tasks run in declared order. Each task's output becomes the next's context. Predictable, traceable, perfect for content pipelines whose order is stable.

crew = Crew(
  agents=[researcher, writer,
          editor, compliance],
  tasks=[research_task, writing_task,
         editing_task, compliance_task],
  process=Process.sequential,
)
result = crew.kickoff()

Lab 8.4 Part 1.

Process.hierarchical

A manager LLM reads the goal and dynamically delegates. Use when the optimal order varies per brief (compliance-first for regulated topics, research-first for trending news).

crew = Crew(
  agents=[researcher, writer,
          editor, compliance],
  tasks=[...],
  process=Process.hierarchical,
  manager_llm="openai/gpt-4o-mini",
)

Part 6

OpenAI Agents SDK
handoffs · guardrails · tracing

Lab 8.5 — the only framework on this list that ships prompt-injection guardrails, PII filtering, voice (RealtimeAgent), and dashboard tracing as first-class primitives.

Three things the SDK gives you for free

① Handoffs as functions

The SDK exposes each handoff target as a tool the LLM can call. You write Agent(..., handoffs=[a1, a2, a3]) — no graph, no edges.

triage = Agent(
  name="Concierge",
  instructions="Route to specialists.",
  handoffs=[account_agent,
            loan_agent,
            investment_agent],
)

② Built-in guardrails

@InputGuardrail runs before the agent; @OutputGuardrail runs after. Each can trip the run with a Pydantic verdict from a dedicated guard agent.

@InputGuardrail
async def detect_injection(ctx, a, x):
    r = await Runner.run(
        injection_detector, x)
    return GuardrailFunctionOutput(
      tripwire_triggered=r.is_injection,
    )

③ Auto-tracing

Every Runner.run_sync() shows up at platform.openai.com → Traces — handoffs, tool calls, guardrails, latency. No instrumentation code.

For multi-provider stacks, you'll still want LangSmith / Langfuse — covered in Module 9.

Part 6 · 1/2

Why "guard agents" beat regex

Regex / keyword filter

BAD = ["ignore previous",
       "system prompt",
       "you are now"]
def is_injection(x):
    return any(b in x.lower()
               for b in BAD)

Brittle. New jailbreak phrasing = miss. Multilingual = miss. Encoding tricks = miss.

Dedicated classifier agent

injection_detector = Agent(
  name="Injection Detector",
  instructions="Classify input...",
  output_type=InjectionCheck,
)
# returns Pydantic
# {is_injection: bool,
#  reasoning: str}

Generalises to novel attacks. Cost: +1 LLM call per turn — fair price for banking.

Lab 8.5 §10 + §11 apply this pattern twice — once for prompt-injection input filtering, once for PII output filtering.

Part 7

Framework selection
decision matrix

The interview-ready answer to "which framework would you pick for our use case, and why?"

The 2-axis quadrant

Reading the quadrant

  • LangGraph — top-left: maximum control, minimal OOTB. You wire everything yourself, but you control everything.
  • MAF — top-right: high control + Azure-managed runtime / identity / monitoring.
  • CrewAI — middle: process types (sequential / hierarchical) trade some control for fast prototyping.
  • OpenAI SDK — bottom-right: handoffs let the LLM decide; guardrails, voice, and tracing are free.
Part 7 · 1/2

One-line decision rules

If the brief says…PickBecause
"State machine with explicit branches, retries, HITL"LangGraphMaximum control + first-class interrupt()
"Org chart of specialists doing creative / research work"CrewAIRole / goal / backstory maps cleanly to job titles
"OpenAI-native + needs voice or PII guardrails out of the box"OpenAI Agents SDKOnly one with built-in guardrails, voice, tracing
"Azure stack, regulated, needs Foundry monitoring"Microsoft Agent Frameworkautogen-agentchat core + Azure-managed runtime
"Compliance-grade audit trail, multi-provider failover"LangGraphProvider-agnostic + checkpointers + LangSmith
"Quick prototype to win the deal next week"CrewAI or OpenAI SDKLowest setup overhead
The frameworks are not zero-sum. Real production stacks often combine: LangGraph as the orchestration backbone, OpenAI SDK guardrails on customer-facing surfaces, MAF for the Azure-hosted parts, CrewAI for marketing-content pipelines.
