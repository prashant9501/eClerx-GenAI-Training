Part 1

Why AWS Bedrock
for regulated banking

Compliance, IAM, VPC, data residency, audit trail — the things that decide whether a model goes to production at a bank.

The five questions a bank asks before any LLM goes live

1. Where does the data go?

Must stay inside the bank's VPC; cannot transit the public internet to a third-party API.

2. Who is allowed to call it?

IAM roles, not API keys. SSO-friendly. Granular per-team permissions.

3. What's the audit trail?

Every call must land in CloudTrail / SIEM the moment it happens.

4. Can we cap blast radius?

Provisioned throughput, per-account quotas, region pinning, kill-switch.

5. Can we enforce safety policies declaratively?

PII masks, denied topics, profanity filters — versioned and testable.

Bedrock answers all five.

That's why it dominates regulated workloads, even though the direct Anthropic API is technically simpler.

Pattern. Prototype against the direct API for speed; before pilot, port the LLM call to ChatBedrock. The chain is unchanged — only the import line moves.
Part 1 · 1/2

The Bedrock service constellation

Bedrock is not a single service — it's a family of APIs sharing one set of model endpoints. Today's labs touch four of them:

  • bedrock — control-plane: list models, create guardrails, manage provisioned throughput
  • bedrock-runtime — data-plane: invoke_model, converse, apply_guardrail
  • bedrock-agent — managed agents (we'll skip in this module)
  • bedrock-agent-runtime — KB retrieve / retrieve_and_generate (Lab 4.4)
LangChain bridge. langchain-aws exposes ChatBedrock + BedrockEmbeddings — same Runnable interface as ChatOpenAI / ChatAnthropic. The pipe operator works unchanged.

Direct API
vs Claude on Bedrock

Same model, two channels. The choice is operational, not capability-driven.

Side-by-side decision table

DimensionDirect Anthropic APIClaude on Bedrock
AuthStatic API keyIAM role + STS
NetworkPublic internet → api.anthropic.comPrivate VPC + PrivateLink
Data residencyAnthropic's regionBedrock region you call
Latency (p50)Lower (one hop)Slightly higher (AWS proxy)
PricingAnthropic list priceSame per-token rate, on AWS bill
Audit / logsAnthropic dashboardCloudTrail + CloudWatch
Throughput SKUAnthropic capacityProvisioned Throughput (committed TPS)
Feature parityLatest (Claude features ship here first)Slight lag (typically days–weeks)
Rule. If the customer is a regulated bank or has VPC requirements → start on Bedrock. If you're prototyping or building a non-regulated SaaS → start on the direct API and port later.
Part 2 · 1/2

Same chain, two LLMs — code-level diff

Direct Anthropic API

from langchain_anthropic import ChatAnthropic

llm = ChatAnthropic(
    model="claude-sonnet-4-5",
    temperature=0.1,
    max_tokens=2048,
)

# rest of the chain is unchanged
chain = prompt | llm | StrOutputParser()

Lower latency · simpler auth · less infra. Pre-pilot default.

Claude on Bedrock

from langchain_aws import ChatBedrock

llm = ChatBedrock(
    model_id="anthropic.claude-sonnet-4-5-20250929-v1:0",
    client=bedrock_runtime,
    region_name=aws_region,
    model_kwargs={"max_tokens": 2048, "temperature": 0.1},
)

# rest of the chain is unchanged
chain = prompt | llm | StrOutputParser()

IAM auth · VPC endpoint · CloudTrail logs · attachable Guardrail. Production default for banks.

The pipe operator and downstream code never change. Picking the channel is a one-line decision — that's the whole reason for using LangChain abstractions over raw SDKs.

Part 3

Bedrock Guardrails
policy as code

Topics, content, PII, regex, profanity — declarative, versioned, attachable to any Bedrock-hosted model.

Where the guardrail sits in a Converse call

One guardrailConfig on a converse call → both input (before the LLM) and output (before reply) are screened. Single network round-trip, single CloudTrail entry.

Five policy types

  • topicPolicyConfig — denied topics with examples
  • contentPolicyConfig — hate / insults / sexual / violence / misconduct
  • sensitiveInformationPolicyConfig — built-in PII + custom regex
  • wordPolicyConfig — denied words + managed profanity list
  • contextualGroundingPolicyConfig — anti-hallucination check

Part 4 · 1/2

Why this shape, not "throw the PDF at Claude"?

Right tool per stage

Textract is faster + cheaper + geometry-aware for layout. Comprehend's PII detector is purpose-built. Claude does what only an LLM can — narrative summary + cross-field reasoning.

PII redaction is a pipeline stage

Comprehend → redact → LLM means PII never lands in your trace store, never appears in cached prompts, never trains a model. The whole stack is auditable.

One function, swappable parts

run_claims_intelligence_pipeline(file) is the contract. Stages are private. Swap Textract for Document AI, Claude for Nova, the calling code never changes.
