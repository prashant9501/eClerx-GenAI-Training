Part 1

From create_agent
to LangGraph

Day 3 you built a single tool-calling agent. Today you build the graph it lives inside — typed state, conditional edges, parallel branches, durable pauses.

What create_agent already gives you

from langchain.agents import create_agent
from langchain_anthropic import ChatAnthropic

llm = ChatAnthropic(model="claude-sonnet-4-5", temperature=0)

agent = create_agent(llm, tools=[search_tool, calc_tool])
result = agent.invoke({"messages": [
    {"role": "user", "content": "What's 2025 GDP of India?"},
]})

A single ReAct-style loop: think → pick tool → call tool → observe → think → answer. The framework handles the loop, retries, error capture.

Strengths

  • Few lines of code
  • Provider-portable tool calling
  • Streaming + callbacks for free

What it doesn't give you

Limits of a single agent loop

  • No typed state beyond the message list
  • No conditional routing to specialist sub-flows
  • No parallel branches with deterministic merge
  • No pause / resume for human approval mid-flow
  • No time-travel replay of a past decision
Module 7 = the moment you outgrow the single loop. LangGraph is the next layer up.
Part 1 · 1/3

When state matters — three signals

① Multiple steps share data

Not just messages — you have a customer_id, a credit_score, an intermediate routing_decision. Every node reads/writes typed fields.

② Routing depends on intermediate output

"If credit_score > 750 and amount < ₹5L, auto-approve; otherwise call HITL." A single agent can't express that without a state machine.

③ The flow can pause for a human

Loan approval, compliance review, escalation gates. interrupt() needs a checkpointer, which needs a state object — which needs a graph.

Heuristic. If you find yourself stashing state inside messages as JSON strings — stop. That's a LangGraph signal.

Part 2

StateGraph
mechanics

Five primitives — TypedDict state, nodes, edges, conditional edges, reducers. Everything else is sugar.

The state — a typed dict with reducers

from typing import Annotated, TypedDict
from langgraph.graph.message import add_messages
import operator

class SupportState(TypedDict):
    # Plain field — last write wins
    customer_id: str
    routing: str

    # Reducer field — append, dedupe by ID
    messages: Annotated[list, add_messages]

    # Reducer field — concat lists from parallel branches
    tool_outputs: Annotated[list[dict], operator.add]

Why TypedDict, not Pydantic

  • Native support for partial updates from nodes
  • Reducers compose with the merge step
  • Pydantic works too — pick when you need strict validation

The reducer is the key idea

ReducerBehaviourWhen
nonelast write winsscalar fields, IDs
add_messagesappend + dedupechat history
operator.addconcat listsparallel writes
custom fn(a,b)your mergesets / dicts / scores
Why this matters. Without reducers, two parallel branches writing the same field would silently overwrite each other. Reducers are the contract between concurrent writers.

Nodes & edges — the topology

from langgraph.graph import StateGraph, START, END

builder = StateGraph(SupportState)

# 1. Nodes — plain Python callables.
builder.add_node("classify",   classify_node)
builder.add_node("billing",    billing_handler)
builder.add_node("technical",  technical_handler)

# 2. Edges — the static topology.
builder.add_edge(START, "classify")
builder.add_edge("billing",   END)
builder.add_edge("technical", END)

# 3. Conditional edge — routes by state.
builder.add_conditional_edges(
    "classify",
    lambda s: s["routing"],          # router fn
    {"billing": "billing", "technical": "technical"},
)

graph = builder.compile()

Part 2 · 2/4

What a node returns — partial state

def classify_node(state: SupportState) -> dict:
    # Read what you need
    last_msg = state["messages"][-1].content

    # Call the LLM (with structured output)
    decision = router_llm.with_structured_output(
        RoutingDecision,
    ).invoke(last_msg)

    # Return ONLY the fields you change.
    # The framework merges them into state via reducers.
    return {
        "routing": decision.category,
        "messages": [AIMessage(
            content=f"Routed to {decision.category}",
        )],
    }

Three invariants you must respect

  • Return a dict of just the fields you changed. Don't return the whole state.
  • For reducer fields, return the delta (e.g. [new_msg]), not the full list. The reducer composes it.
  • For scalar fields, return the new value. The default reducer (overwrite) applies.
Returning {} is legal — the node ran, but the state didn't change. Useful for pure side-effect nodes (logging, observability).
Nodes are pure-ish functions of state → state-delta. That's what makes them composable, testable, and time-travelable.
Part 2 · 3/4

Super-steps — how the runtime executes the graph

LangGraph runs the graph in super-steps. In each super-step:

  1. Find every node whose incoming edges have all fired since the last super-step.
  2. Run them concurrently (threaded for sync, asyncio for async).
  3. Collect each node's partial-state dict.
  4. Merge them through the reducers.
  5. Persist the merged state to the checkpointer.
  6. Schedule the next super-step.
Implication. Two nodes scheduled in the same super-step run in parallel by default. That's how fan-out works — no extra API. (Lab 7.2.)

Part 3

Routing patterns

A classifier node, a routing function, conditional edges. The supervisor pattern in miniature — Lab 7.1 builds the foundation that Module 8 grows into a full multi-agent system.

The router pattern — three pieces

① The schema

Pydantic enum locks the router's output to a finite set of categories. No regex parsing.

class RoutingDecision(BaseModel):
    category: Literal[
      "order_status",
      "returns",
      "billing",
      "shipping",
      "general",
    ]
    confidence: float = Field(ge=0, le=1)

② The classifier node

One LLM call with with_structured_output — typed result lands in state.

def router_node(state) -> dict:
    decision = (router_llm
        .with_structured_output(RoutingDecision)
        .invoke(state["query"]))
    return {
        "routing": decision.category,
        "confidence": decision.confidence,
    }

③ The conditional edge

A pure Python function reads state and returns the next node's name.

def route_by_category(state) -> str:
    return state["routing"]

builder.add_conditional_edges(
    "router",
    route_by_category,
    {"order_status": "order_status_handler",
     "returns":      "returns_handler",
     ...},
)
Lab 7.1 builds this exact pattern for ShopSmart customer support. The same shape becomes the supervisor pattern in Lab 8.1.

Routing variants — three you'll meet in production

PatternRouter signalWhen
Single-axiscategory onlyTopic-based dispatch — simple support routers
Multi-axiscategory + priorityTwo queues per category; VIP fast-path
Confidence-gatedcategory + confidenceBelow threshold → human handoff. The first HITL pattern.
Cycle-permittedspecialist may route back to routerMulti-turn diagnosis — ShopSmart "ask clarifying question" → re-classify
Hierarchicaltop router → sub-routerLarge taxonomies (50+ leaf categories) — supervisor pattern (Module 8)

Anti-pattern · LLM in the routing function

def route_fn(state):
    # ❌ DON'T call an LLM here
    return llm.invoke(...)

The routing function should be cheap & deterministic. LLM calls go in the node, then the function reads its output.

Pattern · LLM in the node, dispatch in the function

def router_node(state):
    # ✅ LLM call here, write to state
    return {"routing": llm_decision(...)}

def route_fn(state):
    # ✅ pure dispatch — fast, deterministic
    return state["routing"]

Parallelisation

Static fan-out for fixed branches. Send API for runtime-many workers. Reducers for deterministic merge. And an empirical cost comparison — because intuition lies.

Static fan-out — known-N branches

builder = StateGraph(ComplianceState)

# 4 fixed compliance checks — known at compile time
for check in ["aml", "kyc", "sanctions", "pep"]:
    builder.add_node(check, make_check_node(check))
    builder.add_edge(START, check)
    builder.add_edge(check, "aggregate")

builder.add_node("aggregate", aggregate_node)
builder.add_edge("aggregate", END)

graph = builder.compile()
# All 4 checks run in parallel in super-step 1.
# Aggregate runs in super-step 2 once all 4 finish.
Reducer: tool_outputs: Annotated[list[dict], operator.add] — each check returns a one-element list, the framework concatenates.
Part 4 · 1/3

Dynamic fan-out — the Send API

What if N depends on the input? An orchestrator-worker shape: a planner node decides how many workers to spawn, with what payload each. Static edges can't express that.

from langgraph.types import Send

def plan_sections(state: ReportState) -> dict:
    # LLM picks 3-7 sections based on the topic
    sections = planner_llm.invoke(state["topic"])
    return {"sections": sections}

def assign_workers(state: ReportState) -> list[Send]:
    # Runtime fan-out — one worker per section
    return [
        Send("write_section", {"section": s})
        for s in state["sections"]
    ]

builder.add_conditional_edges(
    "plan_sections", assign_workers, ["write_section"],
)
Lab 7.2 builds this exact orchestrator-worker. The Send API is what makes Module 8's compliance-report generator work.
Part 4 · 2/3

Cost & latency — what parallelisation actually buys you

~3.0×
typical speedup, 4 parallel calls
1.0×
token cost — identical
max(t_i)
wall-clock floor (vs sum(t_i))

The win

  • Wall-clock latency drops from sum-of-calls to max-of-calls
  • User-facing chat & real-time decisions feel sharper
  • Same input/output tokens — same dollar cost

When it's NOT a win

  • Provider rate limits cap your concurrency
  • Calls < 200 ms — scheduling overhead dominates
  • "Independent" branches that secretly read each other's state
  • 1000+ branches — exhaust the executor pool

Part 5

Human-in-the-loop
& checkpointing

Pause for a human, persist state to disk, replay from any past snapshot. The pattern that turns a demo into a production system.

The HITL primitives — interrupt + Command

from langgraph.types import interrupt, Command

def hitl_gate(state: LoanState) -> dict:
    if (state["amount"] > 25_00_000
        or state["credit_score"] < 750
        or state["ai_confidence"] < 0.85):
        # Pause. State is checkpointed. Process can exit.
        decision = interrupt({
            "for_review": state["draft_decision"],
            "amount":     state["amount"],
            "applicant":  state["customer_id"],
        })
        # When the human resumes, decision is their value.
        return {"final_decision": decision}
    return {"final_decision": state["draft_decision"]}

Resume from any client (CLI, FastAPI, Slack bot):

graph.invoke(
    Command(resume={"approved": True, "officer": "ksingh"}),
    config={"configurable": {"thread_id": "loan-042"}},
)
Part 5 · 1/3

Persistence tiers — pick by deployment shape

CheckpointerStorageRestart-safe?Use when…
InMemorySaverprocess RAM❌Notebooks · demos · CI
SqliteSaverlocal .sqlite file✅ (file)Single-server apps · prototype · trainee labs
PostgresSaverPostgres DB✅ (cluster)Multi-instance prod · regulated workloads · audit trail
Custom (DynamoDB, MongoDB, …)any KV / doc storebackend-dependentAlready standardised on something else

Same API across all three

graph.get_state(config)
graph.get_state_history(config)
graph.update_state(config, values)
graph.invoke(Command(resume=…), config)

Migration is a one-line swap of the constructor — graph code is unchanged.

Common pitfall

Forgetting the checkpointer entirely. Without one, interrupt() has nowhere to save state — the pause becomes a crash. Always compile with checkpointer=… when you have HITL or long-running flows.

Part 5 · 2/3

Time-travel replay — rewind, edit, resume

Every super-step writes a checkpoint. get_state_history walks the list. update_state writes a new checkpoint with edited values. invoke(None, ...) resumes from there.

# 1. Walk history
history = list(graph.get_state_history(
    {"configurable": {"thread_id": "loan-003"}}
))

# 2. Pick a fork point
fork = next(s for s in history
            if "decision_draft" in s.metadata["writes"])

# 3. Write a new checkpoint with edited state
new_config = graph.update_state(
    fork.config, {"ai_confidence": 0.95},
)

# 4. Resume from the edit
graph.invoke(None, config=new_config)

Why this is production magic

  • Reproduce a customer dispute exactly — replay the bad decision step-by-step
  • What-if analysis — change one input, re-run the rest. No re-charging the unchanged LLM calls.
  • Audit trail — every checkpoint is immutable. Regulators love append-only state machines.
  • A/B prompt experiments — fork from the same checkpoint with two prompt variants
Watch out. If your nodes have non-idempotent side effects (charge a card, send an email), replay will repeat them. Use idempotency keys.

Part 6 · 1/2

When agent vs when graph

create_agent — when?

  • The model picks tools freely, no enforced order
  • State is just messages
  • No mid-flow human approval needed
  • You'd rather not think about graph topology

~80% of "agentic" use cases. Q&A bots, research assistants, ad-hoc analysis.

StateGraph — when?

  • Multiple steps need typed shared state
  • Routing depends on intermediate state (not just last message)
  • Mid-flow HITL approval is required
  • Parallel branches with deterministic merge
  • Time-travel replay for audit / debugging
  • Multi-agent supervisor / swarm patterns (Module 8)
You can compose them. A node inside a StateGraph can itself be a create_agent — small loops inside the bigger machine. (Pattern used heavily in Module 8.)

Part 7 · 1/2

Three things to internalise before tomorrow

① State + reducers

If two nodes ever write the same field, you need a reducer. operator.add for lists, add_messages for chat, custom for the rest.

Goal: stop seeing "last write wins" bugs.

② Checkpoint or it didn't happen

Anything with HITL, long flows, or audit needs a checkpointer. SQLite is fine for prototypes; Postgres for prod.

Goal: reach for checkpointer=… by reflex.

③ Agent vs graph

Single loop with free tool choice → create_agent. Typed state, conditional routing, HITL, parallel branches, replay → StateGraph.

Goal: match the abstraction to the problem.

Day 8 (Multi-Agent Systems) builds directly on this — supervisor/swarm/hierarchical patterns are all StateGraphs whose nodes are themselves agents. Bring strong LangGraph muscle memory.
