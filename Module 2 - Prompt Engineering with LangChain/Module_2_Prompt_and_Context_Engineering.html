From prompts to
engineered prompts

A prompt is just code. Once you treat it like code — versioned, tested, structured — you stop fighting the model and start steering it.

The shift: ad-hoc string → engineered artifact

Ad-hoc f-string · Day 1

prompt = f"Summarize this: {text}"
response = client.chat.completions.create(
    model="gpt-4.1-mini",
    messages=[{"role": "user",
               "content": prompt}]
)

Vague task · no persona · no constraints · no schema · output is unpredictable.

Engineered prompt · Day 2

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a senior compliance "
               "analyst. Respond in JSON "
               "with risk_level, reasoning, "
               "and recommendation."),
    ("user", "Summarize this in 3 lines: {text}")
])
chain = prompt | llm | PydanticOutputParser(pydantic_object=Risk)

Persona · output contract · structured · testable · cacheable.

Part 2 · 1/3

MessagesPlaceholder — multi-turn conversations

For chat with history, you need a variable-length slot for prior messages. MessagesPlaceholder is that slot.

from langchain_core.prompts import (
    ChatPromptTemplate,
    MessagesPlaceholder,
)

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a banking assistant."),
    MessagesPlaceholder("history"),     # ← any number
    ("user", "{input}"),
])

# history is a list of HumanMessage / AIMessage
chain.invoke({
    "history": prior_messages,
    "input":   new_user_msg,
})

What goes in history?

  • Previous turns of this conversation
  • Tool outputs (in agent flows)
  • Retrieved context (in RAG)
Pair this with RunnableWithMessageHistory to auto-track per-session history. Module 3 covers the modern memory pattern in depth.
Don't use the deprecated ConversationBufferMemory / ConversationSummaryMemory — they predate LCEL and don't compose with modern chains.
Part 2 · 2/3

FewShotChatMessagePromptTemplate — examples at scale

The problem

Inline string interpolation of 5 examples works. 50 examples? Selectable by similarity? That needs a class.

The solution

FewShotChatMessagePromptTemplate takes:

  • An examples list (or a selector that picks dynamically)
  • An example_prompt — the (user, assistant) sub-template each example uses

Output: a flat sequence of message pairs the model treats as in-context training data.

from langchain_core.prompts import (
    ChatPromptTemplate,
    FewShotChatMessagePromptTemplate,
)

examples = [
    {"input": "transferred ₹50K to TATA",
     "output": "category=Equity"},
    {"input": "ATM withdrawal ₹2K",
     "output": "category=Cash"},
    # ... 100 more
]

example_prompt = ChatPromptTemplate.from_messages([
    ("user", "{input}"),
    ("assistant", "{output}"),
])

few_shot = FewShotChatMessagePromptTemplate(
    examples=examples,
    example_prompt=example_prompt,
)

final_prompt = ChatPromptTemplate.from_messages([
    ("system", "Categorize bank transactions."),
    few_shot,
    ("user", "{input}"),
])
Lab 2.2 §4 builds this with SemanticSimilarityExampleSelector — picks the most-similar 3 of 100 examples per query, so token budget stays tight even with a huge corpus.

Foundational
prompt patterns

The four patterns from Lab 2.2. Master these before reaching for chain-of-thought.

Pattern 1 · Persona

Tell the model who it is and who it's writing for. Goes in the system prompt.

Why it works

  • Activates a domain-specific writing style learned during training
  • Shifts vocabulary, sentence length, refusal threshold
  • Costs almost nothing in tokens

Tuning knobs

  • Role — "senior compliance analyst", "warm bank teller"
  • Audience — "explain to a customer with no banking background"
  • Tone — "calm, factual, no emojis"
SYSTEM:
You are a senior fraud-detection analyst at
a retail bank in India. You write for the
on-call investigations team — concise,
RBI-compliant language, no hedging.

USER:
Investigate transaction #TXN-1234 below
and recommend HOLD or RELEASE.

[transaction details]
Lab 2.2 §1 contrasts neutral persona vs senior-analyst persona on the same prompt — output quality measurably shifts.
Persona is not "playing a character" — it's activating the right slice of the model's training distribution.
Part 3 · 1/4

Pattern 2 · N-Shot prompting

Zero-shot

Just describe the task. Default for capable modern models on common tasks.

Classify sentiment:
"Service was awful."
→

One-shot

Single example to anchor format / style.

"It was great" → positive
"Service was awful." →

Few-shot

2–8 examples. Teaches subtle conventions, niche labels, custom formats.

"It was great" → +
"Could be better" → ~
"Awful service" → -
"Service was awful." →
Practical rule. 3–5 carefully-chosen examples ≈ as good as 50 random ones. Quality > quantity. Lab 2.2 §2 walks through example selection.
Part 3 · 2/4

Pattern 3 · Directional Stimulus

Inject keywords or hints that nudge the model toward the right concept space — without giving away the answer.

Why it works

Keywords activate related concepts via attention. The model "lights up" the relevant region of its knowledge.

When to use

  • Domain-specific summarization (where vocabulary matters)
  • Tasks where the answer is "in the model's head" but it picks the wrong association
  • Nudging refusal-prone models without jailbreaking
# Without stimulus
"Summarize this customer feedback."

# With directional stimulus
"Summarize this customer feedback.
Focus on these themes: pricing, service
quality, app stability, branch hours."
Lab 2.2 §3 demonstrates this on customer-feedback triage — adding 4 hint keywords cuts hallucinated themes by >50%.
Compare to retrieval — stimulus is "free" (no retrieval cost) but only works if the answer is already in the model's parameters.
Part 3 · 3/4

Pattern 4 · Template & Meta-Language

Define a structured input/output convention using delimiters, XML tags, or a custom schema. Then ask the model to fill it in.

Why it works

  • Removes ambiguity about what fields exist
  • Anchors the model's output in your schema
  • Makes downstream parsing reliable

Common conventions

  • JSON — most universal
  • XML tags — Anthropic prefers (training distribution)
  • Markdown headers — for human-readable but parseable
  • Delimited blocks — ###, """
SYSTEM:
You are a banking risk analyst. Output your
analysis in EXACTLY this template:

<risk_assessment>
  <level>low | medium | high</level>
  <reasoning>1-2 sentences</reasoning>
  <recommendation>HOLD | RELEASE</recommendation>
  <confidence>0.0 - 1.0</confidence>
</risk_assessment>

USER: [transaction details]
Templates are the pre-Pydantic way of doing structured output. They still work great with any model. Pydantic + with_structured_output is the modern equivalent — covered in Part 5.

Part 4

Reasoning patterns
CoT · ToT · CoVe

When zero-shot isn't enough, reach here. The patterns from Lab 2.3 — chain, tree, and verification.

Chain-of-Thought (CoT)

Get the model to "show its work" before answering. Reasoning improves on math, multi-step decisions, edge-case detection.

Two flavors

  • Zero-shot CoT — append "Let's think step by step."
  • Few-shot CoT — include 2–3 examples that show the reasoning trace, not just the final answer

Why it works

Each generated step conditions the next — the model effectively gets more "compute" to spread across the answer. Internal token states encode the partial reasoning.

Lab 2.3 §3 uses CoT for fraud-alert analysis & small-business loan eligibility — both cases where the wrong "vibes-only" answer is dangerous.
Part 4 · 1/4

Tree of Thoughts (ToT)

Generalize CoT to multiple parallel branches. The model explores several reasoning paths, evaluates them, picks the best.

Pattern

  1. Generate N candidate next-steps from the current state
  2. Evaluate each (self-critique or scoring)
  3. Prune / expand the most promising
  4. Repeat to depth D, return best path

When it earns its keep

  • Genuine multi-option problems (strategy, planning, design)
  • Tasks where evaluation is cheaper than generation
  • You can budget the extra latency & cost
Lab 2.3 §4 applies ToT to client investment strategy & debt-restructuring planning. Watch the cost — ToT can use 5–10× the tokens of plain CoT.
Part 4 · 2/4

Chain of Verification (CoVe)

A four-step loop that reduces hallucinations by making the model fact-check itself.

  1. Draft — generate an initial answer to the question
  2. Plan verifications — list specific factual sub-questions about the draft
  3. Answer verifications — independently answer each (without seeing the draft)
  4. Revise — generate a final answer reconciling draft + verification results

Why it works

Decoupled verification — the verification step doesn't see the draft, so the model can't anchor its checks to the original (possibly wrong) answer.

Lab 2.3 Part 2 applies CoVe to regulatory-guideline summarization & factual stock analysis — exactly the contexts where a wrong fact has business consequences.
Part 4 · 3/4

When NOT to use these

SituationUseSkip CoT/ToT/CoVe because…
Reasoning model (gpt-5-mini, o1, Claude w/ extended thinking)Just set reasoning_effort="medium"The model already chain-of-thoughts internally; explicit CoT is redundant overhead

Rule: reach for reasoning patterns only when the wrong answer has consequences AND the right answer requires multi-step thought.

Part 5 · 1/4

PydanticOutputParser — the workhorse

from pydantic import BaseModel, Field
from typing import Literal
from langchain_core.output_parsers import PydanticOutputParser
from langchain_core.prompts import ChatPromptTemplate

class LoanApplication(BaseModel):
    applicant_name: str = Field(description="Full legal name")
    amount_requested: float = Field(description="In INR lakhs", ge=0)
    tenure_years: int = Field(ge=1, le=30)
    risk_tier: Literal["low", "medium", "high"]
    flags: list[str] = Field(default_factory=list, description="any concerns")

parser = PydanticOutputParser(pydantic_object=LoanApplication)

prompt = ChatPromptTemplate.from_messages([
    ("system", "Extract a LoanApplication from the customer email.\n"
               "{format_instructions}"),
    ("user", "{email}"),
]).partial(format_instructions=parser.get_format_instructions())

chain = prompt | llm | parser
result: LoanApplication = chain.invoke({"email": email_text})
result.amount_requested  # → float, validated
What this gets you: typed access (IDE autocomplete) · runtime validation (range checks, enums) · automatic JSON-schema generation for the prompt · re-use of the schema in your DB layer / API.
Part 5 · 2/4

.with_structured_output() — the modern path

Most providers now have native structured-output support — they validate the schema on their side and refuse to return invalid JSON. LangChain's .with_structured_output() wraps that.

Differences vs PydanticOutputParser

  • Provider-validated — fewer parse failures
  • No prompt-instruction overhead — schema goes via API, not in tokens
  • One line of code instead of three (parser + prompt + chain)
from pydantic import BaseModel, Field
from langchain_openai import ChatOpenAI

class TriageResult(BaseModel):
    category: Literal["billing", "technical", "fraud"]
    urgency: int = Field(ge=1, le=5)
    summary: str

llm = ChatOpenAI(model="gpt-5-mini")
structured_llm = llm.with_structured_output(TriageResult)

result = structured_llm.invoke(
    "Customer says their card was charged twice "
    "for a single Amazon order on 12 Jan."
)
# → TriageResult instance, validated server-side
Default to this for any new chain in 2026. Fall back to PydanticOutputParser only when running on a model without native support.
Part 5 · 3/4

XML tagging · Decision tree

XML tagging via system prompt

Anthropic's training distribution is XML-heavy. For Claude, XML tags often outperform JSON for "extract structured fields from unstructured text".

SYSTEM:
Extract policy details from the email.
Wrap each field in tags:

<policy_number>...</policy_number>
<coverage_amount>...</coverage_amount>
<effective_date>...</effective_date>
<exclusions>...</exclusions>

Then parse with regex / lxml — no parser library needed.

Which parser?

Need typed object with validation?
├── YES → with_structured_output (Pydantic) ★
└── NO  → what shape?
           ├── single string → StrOutputParser
           ├── list of strings → CommaSeparated
           ├── single date → Datetime
           ├── enum/category → EnumOutputParser
           ├── flexible JSON → JsonOutputParser
           ├── XML (Claude) → XML tagging via system
           └── prone to invalid output?
                → wrap in OutputFixingParser

Prompt caching
across providers

If your prompt has a long stable prefix, you're leaving 50–90% of cost and 4× of latency on the table without caching.

Why cache? — cost & latency math

Without caching

System prompt (5K tokens) + user query (200 tokens) every call.

1M calls/day → 5.2B tokens/day → on Sonnet 4.5: ~$15,600/day.

With caching

System prompt cached → billed at ~10% of normal rate after first hit.

Same 1M calls/day → effective ~$2,000/day.

Savings: 87%.

cheaper · cached input
(Anthropic, OpenAI)
4–5×
faster · TTFT
on cache hit
5 min
default cache TTL
(extendable)
Cache is free money. If your prompt has any reusable prefix (system prompt, RAG instructions, tool definitions, few-shot examples) and you call > 100 times/hour — turn caching on.
Part 6 · 1/5

Anthropic — explicit cache_control

response = anthropic_client.messages.create(
    model="claude-sonnet-4-5",
    max_tokens=1024,
    system=[
        {
            "type": "text",
            "text": "<long bank-policy reference document, ~10K tokens>",
            "cache_control": {"type": "ephemeral"},   # ← cache this!
        }
    ],
    messages=[{"role": "user", "content": "Is X covered under policy Y?"}],
)
# response.usage:
#   cache_creation_input_tokens   (first call)
#   cache_read_input_tokens       (subsequent calls — billed at ~10%)

How it works

  • You mark cache breakpoints with cache_control
  • Anything before the breakpoint is cached
  • Up to 4 breakpoints per request → multi-tier cache
  • Default TTL = 5 minutes (1-hour TTL via beta header)

Pricing

  • Cache creation — 1.25× normal input cost (one-time)
  • Cache read — 0.1× normal input cost (every subsequent hit)
  • Output is unchanged
  • Break-even after ~2 hits
Part 6 · 2/5

Anthropic — multi-layer cache breakpoints

system=[
    {  # Layer 1 — bank policies (3K tokens, near-permanent)
        "type": "text",
        "text": large_policy_doc,
        "cache_control": {"type": "ephemeral"},
    },
    {  # Layer 2 — today's interest rates (500 tokens, refreshes daily)
        "type": "text",
        "text": today_rates_table,
        "cache_control": {"type": "ephemeral"},
    },
    {  # Layer 3 — current session preamble (rarely re-used)
        "type": "text",
        "text": session_metadata,
    },
]
The art of cache design. Stack most-stable on top, most-volatile at bottom. When the bottom layer changes, top layers stay cached. Lab 2.5 §6.1 demonstrates this with multi-tier banking content.
Anti-pattern: putting the user's variable input above a stable reference doc — the cache invalidates on every call. Order matters.
Part 6 · 3/5

OpenAI & Gemini — different shapes, same idea

OpenAI — automatic

No code change required. The platform automatically caches any prompt prefix > 1024 tokens and reuses it across calls within ~5–10 minutes.

  • Cached input billed at ~50% of normal rate (varies by model)
  • You see cached_tokens in response.usage.prompt_tokens_details
  • Stability requires identical prefix (down to whitespace)

Gemini — explicit cache API

You create a named cache resource, then reference it in queries.

cache = client.caches.create(
    model="gemini-2.5-flash",
    config={"contents": large_doc,
            "ttl": "3600s"})

response = client.models.generate_content(
    model="gemini-2.5-flash",
    contents=question,
    config={"cached_content": cache.name},
)
  • Full CRUD — list / get / update / delete
  • Min content size: ~32K tokens (Pro) or 1K (Flash)
  • Storage cost separate from inference

Part 6 · 4/5

Cache-aware prompt design

👎 Cache-hostile

system = (
    f"Today is {datetime.now()}. "    # ← changes every second
    f"User: {user_id}. "              # ← per-user
    f"You are a banking assistant..." # ← stable but BURIED
)

Problem: the variable parts are at the TOP, so nothing before them can be cached → 0% cache hit.

👍 Cache-friendly

system = [
    {"text": LARGE_POLICY_DOC,         # 5K stable
     "cache_control": {"type": "ephemeral"}},
    {"text": LARGE_FAQ,                 # 3K stable
     "cache_control": {"type": "ephemeral"}},
    {"text": f"User: {user_id}. "       # variable
             f"Date: {today}. "}
]

Why it works: stable content first → cache always hits → variable suffix doesn't break the cache above it.

Mental model: cache hits stop at the first byte that differs. Put long-stable stuff first, short-variable stuff last. Lab 2.5 §9 drills this with measurable hit-rate experiments.

Context engineering
the bigger picture

Prompt engineering tunes how the model is asked. Context engineering decides what the model sees. The newer, broader discipline.

Prompt engineering vs context engineering

Prompt engineeringContext engineering
Question"How do I phrase this?""What information should be in the window?"
ScopeThe instructionThe full token budget — instructions + data + memory + tool outputs
LeversPersona · examples · format · CoTRetrieval · summarization · memory · ordering · compression
Failure modes"Vague output""Lost-in-the-middle" · "missed key fact" · "ran out of tokens"

Andrej Karpathy & Tobi Lütke (2024–25): "the new core skill is context engineering — the art and science of filling the context window with just the right information for the next step."
Part 7 · 1/3

The 5 information layers in a context window

Each layer has different sourcing, different lifespan, different cost. Treating them as one giant string is the #1 cause of wasted cost and lost-in-the-middle errors.
Part 7 · 2/3

The context budget — allocation framework

Step 1 · Set the budget

Pick a target — e.g. 32K input tokens (out of 200K window). Reserves the rest for output and headroom.

Step 2 · Allocate by layer

  • System / few-shot: 5K (cached, ~$0)
  • RAG chunks: 15K (5–10 chunks, top-k)
  • Memory: 8K (summary + last 5 turns)
  • Current turn: 4K (input + tool outputs)

Step 3 · Measure & tune

Log per-call: tokens by layer, hit rate, latency. Re-balance when one layer crowds out another.

Ratios change per use case — code agents skew RAG-heavy, customer-support skews memory-heavy. Always measure first, never assume.

Part 8

Context management
techniques

When the context window starts to overflow — and it always does — these are the patterns that keep your application alive.

Truncation — the simplest move

Head truncation

Drop oldest tokens first. Keeps recent context.

Use for: chat where recent turns matter most.

Risk: loses original system prompt & goals.

Tail truncation

Drop newest tokens. Keeps original setup.

Use for: rare — only when system prompt is the most important.

Risk: ignores user's actual question.

Head + tail (default) ★

Keep system prompt & most recent N turns; drop the middle.

Use for: production chat.

Best balance — but middle context is gone forever.

Part 8 · 1/5

Summarization — running & hierarchical

Running summary

After every k turns, summarize the older portion into a compact paragraph. Replace it in the context window.

if len(history) > 10:
    old, recent = history[:-5], history[-5:]
    summary = llm.invoke(
        f"Summarize this chat in 200 words: {old}"
    )
    history = [summary] + recent
Trade-off: cheaper context, but summary loses fidelity each pass. Compounds over a long session.

Hierarchical summarization

Multiple levels — last-5-turns summary, last-50-turns summary, last-500-turns summary. Older = more compressed.

Part 8 · 2/5

Sliding-window memory

Keep the last N tokens verbatim. Anything older falls off the edge.

When it works

  • Tasks where only recent context matters (short-form chat, code completion)
  • Stateless services on top of stateful clients
  • Cost-sensitive deployments — bounded context = bounded cost

When it fails

  • User refers to something said 30 turns ago — gone
  • Multi-day support cases — needs persistent memory below the window

Part 8 · 4/5

Lost-in-the-middle & ordering

Models recall information at the start and end of long contexts much better than in the middle. Documented across GPT-4, Claude, Llama, Gemini.

Mitigations

  • Reorder by relevance — most-relevant chunks first AND last
  • Re-rank before insertion — cross-encoder bumps top hits up
  • Compress mid-context — summarize less-relevant chunks
  • Recency anchor — restate the question right before the answer position
  • Smaller windows — fewer chunks → less middle to lose
Practical: when doing RAG with k=10 chunks, put your top-3 most-relevant at the start, the next 3 at the end, and the rest in the middle. Module 5 covers this in depth.
