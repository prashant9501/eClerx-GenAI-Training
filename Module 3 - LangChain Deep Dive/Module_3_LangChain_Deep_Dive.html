Part 1 · 1/3

The Runnable interface — the unit of composition

Every LangChain component (prompt, LLM, parser, retriever, lambda) is a Runnable. They all share the same interface — that's why the pipe operator works.

The 5 methods you'll use

MethodReturnsWhen
.invoke(x)single resultone call
.stream(x)iterator of chunkstoken-stream UI
.batch([x1, x2])list of resultsparallel calls
.ainvoke(x)awaitableasync server
.astream_events(x)fine-grained eventstracing / SSE
Mental model. If you can write f(g(h(x))), you can write h | g | f. The pipe just spells composition left-to-right.
Part 1 · 2/3

What changes structurally on Day 3

① From f-strings to chains

You stop writing response = llm.invoke(prompt) by hand. You write chain = prompt | llm | parser and call chain.invoke(...).

② From "no memory" to per-session memory

You wrap the chain in RunnableWithMessageHistory. Each user gets their own conversation thread, identified by session_id.

③ From "answer text" to "stream events"

.stream() for token UIs. .astream_events() for fine-grained traces — the basis of M9 observability and M10 SSE endpoints.

④ From "one path" to "many paths"

RunnableParallel, RunnableLambda, RunnableBranch, .with_fallbacks() — branch, fan-out, recover.

⑤ From "fixed steps" to "agent decides"

The model picks which tool to call. create_agent (LangChain v1) replaces the legacy AgentExecutor orchestration.

⑥ From "happy path" to "production loop"

Tool failures, wrong-tool selection, infinite loops, timeouts. Module 3 ends with the debugging mindset.

Part 2

LCEL composition
patterns

The five primitives you'll reach for daily — pipe, parallel, passthrough, lambda, branch.

The pipe — prompt | llm | parser

The simplest LCEL chain. The right-hand side's .invoke() consumes the left-hand side's output. Same shape, regardless of how many stages.

from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_anthropic import ChatAnthropic

prompt = ChatPromptTemplate.from_messages([
    ("system", "You are a banking risk analyst."),
    ("user", "Assess: {transaction}"),
])
llm = ChatAnthropic(model="claude-haiku-4-5", temperature=0)
parser = StrOutputParser()

chain = prompt | llm | parser
risk_summary = chain.invoke({"transaction": tx_json})

What you also get for free

  • .stream({"transaction": ...}) — token-by-token
  • .batch([{"transaction": t1}, {"transaction": t2}]) — parallel
  • chain.with_config({"tags": ["risk"]}) — for tracing

What the pipe is doing under the hood

# Conceptually:
def chain_invoke(input_dict):
    prompt_value = prompt.invoke(input_dict)
    ai_message   = llm.invoke(prompt_value)
    text         = parser.invoke(ai_message)
    return text

# LCEL just wraps that, but also gives you
# stream / batch / async / tracing / fallbacks
# without having to rewrite the function.

Part 2 · 1/4

RunnableParallel — fan-out, fan-in

Run multiple Runnables on the same input concurrently. The result is a dict keyed by your branch names.

from langchain_core.runnables import RunnableParallel

# Three perspectives on the same email
analysis = RunnableParallel({
    "summary":      summary_chain,
    "sentiment":    sentiment_chain,
    "action_items": action_chain,
})

result = analysis.invoke({"email": email_text})
# result == {
#   "summary":      "...",
#   "sentiment":    "...",
#   "action_items": [...],
# }

Each branch runs concurrently via the LangChain executor (threaded for sync, asyncio for async). For 3 branches → ~1× latency instead of 3×.

Lab 3.1 §8 uses RunnableParallel for a 3-branch document analysis. Module 7 generalises this pattern with the LangGraph Send API.
Part 2 · 2/4

RunnableLambda + RunnablePassthrough.assign

RunnableLambda

Wrap any Python callable into the Runnable interface — so it composes via |.

from langchain_core.runnables import RunnableLambda

def normalize(text: str) -> str:
    return text.strip().lower()

clean = RunnableLambda(normalize)
chain = clean | prompt | llm | parser

When use it

  • Pre / post-processing in the middle of a chain
  • Calling a non-LangChain library (regex, datetime, API wrapper)
  • Custom transforms that don't justify a full Runnable subclass

RunnablePassthrough.assign

Add fields to a dict-shaped pipeline state without dropping the existing ones. Used everywhere in RAG.

from langchain_core.runnables import RunnablePassthrough

# Input:  {"question": "..."}
# Add a "context" field, keep "question"
chain = (
    RunnablePassthrough.assign(
        context=lambda x: retriever.invoke(x["question"])
    )
    | prompt   # uses both {question} and {context}
    | llm
    | parser
)
Lab 3.1 §7 + Module 4 RAG labs lean on this pattern. It's the LCEL-idiomatic way to build pipeline state.
Part 2 · 3/4

Branching — RunnableBranch, fallbacks, retries

RunnableBranch

If/elif/else routing on the input. Use when the routing rule is deterministic (regex, type check, length).

RunnableBranch(
  (lambda x: x["type"]=="urgent",
   urgent_chain),
  (lambda x: x["type"]=="vip",
   vip_chain),
  default_chain,
)

For LLM-based routing, use a small classifier chain (Module 7).

.with_fallbacks(...)

If the primary Runnable raises, try the next one. Best for provider failover.

primary = ChatAnthropic(...)
backup  = ChatOpenAI(...)

resilient = primary.with_fallbacks(
    [backup]
)

Fires on any exception. Doesn't retry the same provider.

.with_retry(...)

Retry the same Runnable on failure. Best for transient errors (rate limits, timeouts).

llm.with_retry(
  retry_if_exception_type=(
    RateLimitError,
  ),
  stop_after_attempt=3,
  wait_exponential_jitter=True,
)

Combine with fallbacks: retry first, then fail over.

Layered defence: llm.with_retry(...).with_fallbacks([backup]) — exponential-backoff against the primary, then switch providers if it still fails. This is the production pattern.

Part 3

Modern memory
RunnableWithMessageHistory

The deprecated ConversationBufferMemory family is gone in 1.0. The new pattern composes cleanly with LCEL and scales to multi-tenant deployments.

Why ConversationBufferMemory was deprecated

Legacy memory · pre-LCEL

from langchain.memory import (
    ConversationBufferMemory,
)
memory = ConversationBufferMemory(
    memory_key="history",
    return_messages=True,
)
chain = LLMChain(
    llm=llm, prompt=prompt, memory=memory,
)
out = chain({"input": "hi"})  # memory side-effected

Stateful object embedded in a Chain · single-process · bound to the chain instance · doesn't compose with LCEL pipes · per-user requires N memory objects.

Modern memory · LCEL-native

from langchain_core.runnables.history \
    import RunnableWithMessageHistory
from langchain_core.chat_history \
    import InMemoryChatMessageHistory

store = {}
def get_history(session_id: str):
    if session_id not in store:
        store[session_id] = InMemoryChatMessageHistory()
    return store[session_id]

chain_with_mem = RunnableWithMessageHistory(
    chain, get_history,
    input_messages_key="input",
    history_messages_key="history",
)

Pure wrapper · composes with any Runnable · per-session via session_id · history backend is pluggable (in-mem, SQLite, Postgres, Redis).

The mental shift. Memory is no longer a property of the chain — it's a wrapper that injects history before the call and saves the new turn after. The chain stays stateless and reusable.
Part 3 · 1/3

How RunnableWithMessageHistory actually works

The session_id

Passed via the configurable key in the run config. Every call you make includes it; the wrapper uses it to look up the right history.

chain_with_mem.invoke(
    {"input": "hi, I'm Priya"},
    config={"configurable": {"session_id": "u-123"}},
)
chain_with_mem.invoke(
    {"input": "what's my name?"},
    config={"configurable": {"session_id": "u-123"}},
)
# → "Your name is Priya."

Why this scales

  • Stateless chain — one instance, N concurrent users
  • History store can be Redis / Postgres → multi-process
  • session_id = your user_id, conversation_id, or both

Part 4

Streaming
.stream · .astream_events

For UIs you want token streaming. For tracing, observability, and SSE endpoints you want event streaming. They're different APIs for different jobs.

Three ways to consume a chain's output

MethodWhat you getLatencyWhen use
.invoke(x)Final result, all at onceWait for full outputBatch jobs · post-processing
.stream(x)Iterator of output chunks (text deltas, dict diffs)First token in < 1sChat UIs · token streaming
.astream_events(x, version="v2")Iterator of fine-grained events: chain start, llm token, tool call, endFirst event immediatelySSE endpoints · LangSmith-like tracing · debug

.stream — for users

for chunk in chain.stream({"q": q}):
    print(chunk, end="", flush=True)
# "The" "loan" " application" " indicates" ...

.astream_events — for the platform

async for ev in chain.astream_events(
    {"q": q}, version="v2"):
    if ev["event"] == "on_llm_stream":
        ...   # send to SSE
    elif ev["event"] == "on_tool_end":
        ...   # log audit trail
Lab 3.1 §6 walks through both. Module 10 wires astream_events into a FastAPI StreamingResponse for the deployed agent — that's why it matters.

Part 5

Tools & agents
the v1 path

A chain runs a fixed sequence. An agent picks its own next step. create_agent (LangChain v1) replaces the legacy AgentExecutor — leaner, async-native, message-shaped.

The agent loop — what's actually happening

An agent is just a chain in a while-loop. The model returns either a final answer or a tool call. If it's a tool call, you run the tool, append the result, and ask the model again.

  1. Model receives messages + tool schemas
  2. Model returns either:
    • final answer (we stop), or
    • tool_calls array (we run them)
  3. Append ToolMessage with each result
  4. Loop back to step 1

Stopping conditions

  • Model returns final text (no tool calls)
  • Hit max iteration cap (default 25 in create_agent)
  • Hit timeout / cancellation
Part 5 · 1/4

Defining tools — @tool vs StructuredTool

@tool — the quick form

Decorate a typed function. The docstring becomes the LLM-visible description; type hints become the schema.

from langchain_core.tools import tool

@tool
def get_balance(account_id: str) -> float:
    """Return the current INR balance for the
    given bank account_id. Use only for
    customer-owned accounts."""
    return banking_db.balance(account_id)

When use it

  • Schema is simple (1–3 primitive args)
  • Docstring describes the contract well
  • You don't need custom validation

StructuredTool + Pydantic — strict form

Define an explicit Pydantic input schema. Get validation, descriptions per field, enums, ranges.

from pydantic import BaseModel, Field
from langchain_core.tools import StructuredTool

class TransferInput(BaseModel):
    from_account: str = Field(description="...")
    to_account:   str = Field(description="...")
    amount_inr:   float = Field(gt=0, le=200_000)
    note:         str | None = None

def _transfer(from_account, to_account, amount_inr, note):
    ...

transfer = StructuredTool.from_function(
    func=_transfer,
    name="transfer_funds",
    description="Move INR between two own accounts.",
    args_schema=TransferInput,
)
Pick this for anything non-trivial or dangerous (transfers, deletes, external API writes).
Part 5 · 2/4

Manual loop with bind_tools — and what's missing

Useful to see once, then never write again. bind_tools tells the LLM about tools but you do all the loop bookkeeping by hand.

llm_with_tools = llm.bind_tools([get_balance, transfer])

messages = [HumanMessage(content="check balance for A1")]
while True:
    ai = llm_with_tools.invoke(messages)
    messages.append(ai)
    if not ai.tool_calls:
        break  # final answer
    for call in ai.tool_calls:
        result = TOOLS[call["name"]].invoke(call["args"])
        messages.append(ToolMessage(
            content=str(result), tool_call_id=call["id"]
        ))

What this loop is missing

  • Iteration cap — runaway loops
  • Per-tool error handling — one bad call kills it
  • Parallel tool execution — N calls run sequentially
  • Streaming — no partial visibility
  • Tracing hooks — no LangSmith / Langfuse out of the box
  • Cancellation / timeout — no clean shutdown
Every one of these is a real bug Lab 3.2 §8 walks through. Then §9 introduces the solution: create_agent.
Part 5 · 3/4

create_agent — the LangChain v1 modern path

from langchain.agents import create_agent
from langchain_anthropic import ChatAnthropic

llm = ChatAnthropic(
    model="claude-sonnet-4-5",
    temperature=0,
)

agent = create_agent(
    model=llm,
    tools=[get_balance, transfer, get_rate],
    prompt="You are a banking assistant. "
           "Use the tools to answer accurately.",
)

result = agent.invoke({
    "messages": [{
        "role": "user",
        "content": "What's my A1 balance, and "
                   "transfer ₹5K to A2?",
    }]
})
final = result["messages"][-1].content

What you get vs the manual loop

  • Iteration cap built-in (default ~25)
  • Parallel tool execution where possible
  • Streaming via .astream(...)
  • Native LangSmith / callback integration
  • Per-tool error injection back to the LLM
  • Async-first — drop into FastAPI directly

vs legacy AgentExecutor

  • Message-shaped state (matches LangGraph)
  • Composes with RunnableWithMessageHistory for chat
  • One import, no create_*_agent + AgentExecutor two-step
Lab 3.2 §9 rewrites the manual loop using create_agent. Side-by-side it's ~⅓ the code and you stop hand-rolling bookkeeping.

Part 6

Production
concerns

Once you ship an agent to real users, the failure modes that don't show up in notebooks are the ones that page you at 2am.

Errors inside the tool loop

The agent calls get_weather("Mumbai"). The API is down. What now? Three options, only one is right.

StrategyUXVerdict
Crash the whole agent500 error❌ silly
Silently skip the tool"I don't know"❌ deceptive
Inject error back into messagesAgent retries / picks alternate✅ correct

create_agent does the third by default — exception → ToolMessage(content="error: ...", status="error"). The next LLM step sees it and adapts.

# Custom error handler at the tool level
@tool
def get_weather(city: str) -> str:
    """Get current weather for an Indian city."""
    try:
        return weather_api.fetch(city)
    except WeatherAPIDown:
        return ("ERROR: weather API is down. "
                "Tell the user to retry in a minute.")
    except CityNotFound:
        return f"ERROR: '{city}' not found. Suggest "
               "nearest major city."

# Now the LLM sees readable errors and routes around them.
The trick: errors that read like instructions (because they go to the LLM) almost always result in the agent recovering gracefully. Errors that read like stack traces don't.
Part 6 · 1/3

Debugging — when the agent picks the wrong tool

① Read the trace

Print every tool_call and the messages in scope. The wrong choice almost always becomes obvious — an irrelevant tool description, a missing arg.

② Sharpen tool descriptions

Treat the docstring as a prompt for the LLM. State when to use AND when not to use. Concrete examples beat abstract verbs.

③ Reduce tool count

10 tools → 30% wrong-tool rate. 4 tools → ~5%. Group related tools, remove unused ones, or split by sub-agent (Module 8).

④ Add few-shot examples to the prompt

"For balance questions, use get_balance; for rate questions, use get_rate" — explicit routing examples in the system prompt.

⑤ Tighten arg schemas

Wrong-arg ≠ wrong-tool, but feels the same. Pydantic Literal, conint(ge=...), descriptive field text — all reduce the model's guesswork.

⑥ Lower temperature

Default to temperature=0 for the agent's planner. Creativity is for the response, not the routing.

Lab 3.2 §11 intentionally provokes a wrong-tool selection, then walks through fixes 1–6 above. It's the most-cited interview-prep section of the day.

Forward references — where this leads

Module 7 · LangGraph

The create_agent loop is great for "model + tools." When you need explicit state, branches, parallel branches, HITL, and checkpoints, you graduate to LangGraph.

Same agent, but now you can pause for human approval, persist state across crashes, and branch based on results.

Module 9 · Observability

Custom BaseCallbackHandler hooks expose every on_llm_start / on_tool_end event for audit logging. LangSmith automates most of this.

Today's astream_events is the read-only variant. The callback API is the write side.

Module 10 · Deployment

agent.astream_events() → FastAPI StreamingResponse → SSE stream → browser. The pattern that turns a notebook agent into a public-URL service.

Today is the streaming foundation; M10 is the deployment story.
