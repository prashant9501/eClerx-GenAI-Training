Part 1

Why RAG?

Hallucinations, stale training data, and unauditable answers — the three problems RAG solves that fine-tuning doesn't.

Three problems with raw LLM Q&A

① Hallucination

Even SOTA models confidently invent facts about niche or private domains they weren't trained on. Bank-internal policy > the model's prior knowledge.

② Knowledge cutoff

Training data freezes ~6–12 months pre-release. Rate cards, regulatory updates, and product launches after that date are invisible to the model.

③ Unauditable

"The interest rate is 12.5%" — but says where? Compliance and legal teams need traceability: which doc, which page, which clause.

RAG's bet. Don't retrain the model on your private docs — instead, retrieve the relevant chunks at query time and stuff them into the prompt. Cheaper, faster to update, and citations come for free.
Part 1 · 1/3

The canonical RAG pipeline

Two phases. Ingest runs once (or on doc updates) — heavy. Query runs on every user request — must be fast. Today's labs build both.
Part 1 · 2/3

RAG vs Fine-tuning — when use which?

Fine-tuning

  • Training pass on your data → updated weights
  • Best when: behaviour, style, or output format must change
  • Cost: GPU hours, ML-ops, eval pipeline
  • Updates: new training run per refresh
  • Citations: ❌ — knowledge baked into weights

RAG

  • No training — knowledge stays in retrievable docs
  • Best when: facts/policy/rate-card change frequently
  • Cost: embeddings + vector-store hosting + per-query LLM
  • Updates: re-embed only the changed docs
  • Citations: ✅ native — chunk metadata = source
Default to RAG. 90% of "use my company data with an LLM" use cases are RAG. Fine-tuning is a tool of last resort: brand voice that prompting can't capture, output formats that few-shot can't lock, or a model that needs to reason about a domain in a way no prompt can elicit.

Part 2 · 1/4

Why chunk at all?

Embedding context limits

Most embedding models cap at 512–8K tokens. A 100-page PDF becomes one blurry vector — useless for retrieval.

Retrieval focus

Smaller chunks → tighter, more topical embeddings → better precision. Top-k of 3 small chunks beats top-k of 1 huge chunk.

LLM context budget

You inject top-k into the prompt. Small chunks let you fit more diverse evidence; large chunks burn the budget on one document.

The trade-off you're tuning. Too small → fragments lose context, retrieval gets noisy. Too large → embeddings blur, prompt fills up fast. Sweet spot for prose RAG: 500–1000 tokens with 10–15% overlap. Always measure on your eval set.
Part 2 · 2/4

Chunk overlap — what and why

Why overlap. "Form 16" sits at a chunk boundary. Without overlap, chunk 2 has the phrase, chunk 3 doesn't — a query about Form 16 might miss either. Overlap = each boundary phrase appears in both adjacent chunks. Standard 10–15% of chunk_size.

Part 3

Embeddings
& the geometry of meaning

Why is "high-yield savings account" close to "term deposit" but far from "credit card late fee"? It's all geometry.

What an embedding is

An embedding is a fixed-length vector — typically 384, 768, 1024, or 1536 dimensions — produced by a model trained so that semantically similar texts produce vectors with similar direction.

from langchain_openai import OpenAIEmbeddings

embed = OpenAIEmbeddings(model='text-embedding-3-small')
vec = embed.embed_query('home loan interest rate')
# → list of 1536 floats; each one a learned latent feature
  • Direction carries meaning
  • Magnitude mostly carries length / token count
  • Models are trained with contrastive loss: similar pairs pulled together, dissimilar pushed apart
Part 3 · 1/3

Three similarity metrics — when do they agree?

Cosine similarity

Angle between vectors only — magnitude ignored. Default for embeddings because magnitude often correlates with text length, not semantics.

cos(a,b) = a·b / (‖a‖·‖b‖)

Euclidean distance

Straight-line distance — penalises both direction and magnitude differences. Longer text → larger vector → more euclidean distance even if same topic.

Dot product

Equals cosine when both vectors are pre-normalised (Titan v2 with normalize=True, OpenAI by default). Faster than cosine since no division.

Practical rule. All three are monotonically equivalent for L2-normalised vectors — they return the same top-k. Most embedding models normalise. Default to cosine; switch to dot product when the vector store offers it (faster); use euclidean only when magnitude actually means something.

Vector stores

FAISS in-memory vs ChromaDB persistent vs OpenSearch — and the indexing strategies that scale to billions.

FAISS vs ChromaDB — when use each

Default rule. Start with FAISS for prototypes and static corpora. Upgrade to ChromaDB (or pgvector / OpenSearch / Pinecone) when you need metadata filtering, multi-tenant isolation, or the corpus updates often.

Part 4 · 2/3

Metadata filtering — the production multiplier

Naive: post-filter

Retrieve top-50 by similarity → drop everything that doesn't match version=2024.3.

Problem: if only 1 of your top-50 matches, you've wasted 49 retrievals and may return zero results to the user.

Vector-store-side filter

chroma.similarity_search(
    query=q, k=3,
    filter={'version': '2024.3'},
)

Filter applies before the vector search — your top-k comes from the eligible subset only.

Critical for. Tenant isolation (multi-customer KBs) · version control · permission scoping (PII docs visible only to authorised users). This is the headline reason to reach for Chroma / pgvector / OpenSearch over FAISS — FAISS doesn't support metadata filters natively.

Part 5

End-to-end RAG
with LCEL

Compose retriever + prompt + LLM + parser into one runnable. Return citations with every answer.

The LCEL RAG pattern

from langchain_core.runnables import (
    RunnableParallel, RunnablePassthrough,
)

retriever = vectorstore.as_retriever(
    search_kwargs={'k': 3}
)

def format_docs(docs):
    return "\n\n".join(
        f"[doc {i+1}] {d.page_content}"
        for i, d in enumerate(docs)
    )

rag_chain = (
    {"context": retriever | format_docs,
     "question": RunnablePassthrough()}
    | rag_prompt
    | llm
    | StrOutputParser()
)
result = rag_chain.invoke("What's the prepayment fee?")
Same LCEL primitives from Module 3 — the chain composition story doesn't change. Retriever joins as just another Runnable.
Part 5 · 1/2

Citations — return docs alongside the answer

rag_with_sources = RunnableParallel(
    {"docs": retriever,
     "question": RunnablePassthrough()}
).assign(
    answer=(
        {"context": lambda x: format_docs(x["docs"]),
         "question": lambda x: x["question"]}
        | rag_prompt
        | llm
        | StrOutputParser()
    )
)
result = rag_with_sources.invoke("...")
# result['answer']  → str
# result['docs']    → list[Document]
#   each with metadata['source'], metadata['page']

Wrap the chain in RunnableParallel so two branches run from the same input — one builds the answer, the other passes the retrieved Documents through.

Why citations matter

  • Auditability — compliance and legal teams need traceability for every claim
  • Debuggability — wrong answer? you immediately see the retrieved doc that misled the LLM
  • UI — citations render as footnotes / clickable links so the user can verify any claim
  • Hallucination signal — answer claims a fact NOT in any retrieved doc → grounding failure → tune retrieval
Production tip. Always log result['docs'] alongside the answer to your trace store. When users complain, you replay with the exact retrieved context.

Managed RAG

Same RAG pipeline you just built — but AWS owns chunking, embedding, vector store, and retrieval. Build vs buy.

Bedrock Knowledge Bases pipeline
