Part 1

The Transformer

The single architectural idea (2017) that made every modern LLM possible. Replaces sequential recurrence with parallel attention.

The pre-transformer bottleneck

Before transformers, sequence models were recurrent: process token 1, update hidden state, process token 2, repeat.

  • RNN Vanishing gradients on long sequences.
  • LSTM Gates fix vanishing — but still sequential, can't parallelize over tokens.
  • Seq2Seq + attention (2014–15) Encoder-decoder for translation; attention added as a patch.

The bottleneck

Every token had to wait for the previous one's hidden state. Training a 1B-parameter RNN at scale was practically impossible — too slow on GPUs.

Then someone asked: do we need recurrence at all?
Part 1 · 1/4

2017 · "Attention Is All You Need"

Vaswani et al. (Google Brain) — the paper that started everything.

Key insight

Replace recurrence with self-attention: every token attends to every other token in parallel.

Why it mattered

  • Parallelizable — finally fits GPU compute
  • Long-range — direct token-to-token paths, no vanishing
  • Scales — bigger model + more data → predictably better, with no architectural ceiling yet found
  • Universal — same architecture now powers vision, audio, video, code

The big numbers

  • 2017 — original transformer · 65M params
  • 2018 — BERT-large · 340M
  • 2020 — GPT-3 · 175B (×1000)
  • 2024 — Llama 3.1 405B (open) · GPT-4 estimated > 1T
Same core architecture, scaled 10,000× in 7 years.
Part 1 · 2/4

The transformer block — anatomy

The two operations per block

1 · Multi-head attention — tokens look at each other (mixing across positions).

2 · Feed-forward network — each token's representation is transformed independently (mixing across features).

Residuals + LayerNorm wrap both operations — they make deep stacks trainable. Without them, gradients die after a few layers.
Repeat the block N times. GPT-3: 96 blocks. Llama 3.3 70B: 80 blocks. Each block adds capacity to learn more abstract patterns.
Part 1 · 3/4

Stacking layers — depth = capacity

Stacking transformer blocks creates a hierarchy of representations:

  • Early layers — surface features, syntax, local patterns
  • Middle layers — phrase- and sentence-level structure
  • Late layers — semantics, topic, task-specific features
Same architecture, different depth → different model classes. Width (hidden dim) and depth (num layers) both matter.

Part 2

Three architecture
families

All transformer-based — but the attention mask and training objective differ. Each is purpose-built for a different task class.

Encoder-only — bidirectional understanding

Examples

BERT RoBERTa DistilBERT DeBERTa

How it sees text

Every token attends to every other token in both directions. The model has full context before predicting anything.

Best for

  • Classification (sentiment, intent, spam)
  • Token-level tasks (NER, POS tagging)
  • Embeddings & retrieval (semantic search, RAG)
  • Extractive Q&A (find span in passage)
Trained via masked-language-modeling. Hide 15% of input tokens, predict them — forces bidirectional understanding.
Part 2 · 1/4

Decoder-only — autoregressive generation

Trained to predict the next token. Causal mask forbids "looking ahead". Same model can therefore generate, autoregressively, at inference time.

Examples

GPT family Claude Llama Mistral Gemini DeepSeek

How it generates

Autoregressively — predict next token, append, predict next, repeat until EOS or max_tokens.

Best for

  • Open-ended text generation
  • Chat / conversation
  • Code generation, completion, refactoring
  • Reasoning & planning
  • Tool use and agents
The dominant paradigm. All frontier LLMs (GPT-5, Claude, Gemini, Llama) are decoder-only.
Part 2 · 2/4

Encoder-decoder — sequence-to-sequence

Examples

T5 FLAN-T5 BART mT5

How it works

Two stacks bridged by cross-attention:

  • Encoder reads the full input bidirectionally
  • Decoder generates output autoregressively, attending to encoder outputs

Best for

  • Translation
  • Summarization
  • Structured rewriting (text-to-text)
Modern very-large decoder-only models can do most enc-dec tasks just by being prompted — so encoder-decoders are declining in popularity, but still strong for small / on-prem / fixed-task deployments.

Part 3

Self-attention
deep dive

The single mechanism that powers every modern LLM. We'll keep math to a minimum and focus on intuition you can describe in an interview.

Query · Key · Value — the intuition

For each token, the model produces three vectors via learned linear projections:

Query (Q)

"What am I looking for?"

Key (K)

"What do I represent / advertise?"

Value (V)

"What information do I carry?"

Attention = a weighted retrieval: my Q against everyone's K gives weights, then I take a weighted sum of their Vs.

Part 3 · 1/4

The attention formula

Attention(Q, K, V) = softmax( Q · Kᵀ / √dk ) · V
Vaswani et al., 2017 — the equation that runs every modern LLM

Q · Kᵀ

Dot products between every query and every key. Output: an n × n attention-score matrix (n = seq length).

÷ √dk

Scaling — keeps dot products from growing too large at high embedding dimensions and saturating softmax.

softmax(...)

Normalizes each row to a probability distribution (each query's attention weights sum to 1).

· V

Mix the values according to the attention weights — produces each token's new representation.

The cost. Q·Kᵀ is n × n in sequence length — quadratic. This is why long-context is expensive (Part 6).
Part 3 · 2/4

Multi-head attention — parallel views

Instead of one big attention computation, run h heads in parallel with smaller Q/K/V projections, then concatenate and project back.

  • Each head learns a different relation — syntax, coreference, long-range topical links
  • Compute is the same — h smaller heads ≈ one big head
  • More expressive — model can attend to multiple things at once
Typical settings. BERT-base = 12 heads · GPT-3 = 96 heads · Llama 3 8B = 32 heads. Head count usually scales with hidden dimension.
Part 3 · 3/4

What heads & layers learn

Layers — top to bottom

  • Early (layers 1–4) — local syntax, adjacent-word relations
  • Middle (layers 5–8) — phrase- & clause-level structure
  • Late (layers 9–12+) — semantics, topic, task-specific

Empirically observed in BERT probing studies (Tenney et al., 2019; Clark et al., 2019).

Heads — within a layer

  • Some heads attend strongly to previous token (a "shift register")
  • Some heads attend to matching brackets / quotes
  • Some heads attend to coreferent pronouns
  • Many heads are redundant — pruning them often barely affects accuracy
Lab 1.4 §9 visualizes average attention across heads in the last layer of BERT for a banking sentence. You'll see "loan" attending to "approved" and "bank" — exactly what semantic late-layer attention looks like.
Attention heatmaps are a practical interpretability tool — when a model misclassifies, the heatmap often shows it attended to the wrong tokens.

Part 4

Tokenization

LLMs don't see words — they see tokens. The unit of cost, the unit of context, and a frequent source of "why is this prompt so expensive?" surprises.

Why tokens — not characters or words?

Characters

Pros: tiny vocabulary, no OOV.

Cons: sequences become 4–5× longer · loses word-level structure · poor compute efficiency.

Words

Pros: natural unit, intuitive.

Cons: huge vocab (millions for multilingual) · OOV problem (any new word breaks) · poor for compounds & agglutinative languages.

Subwords ✓

Pros: ~30K–200K vocab fits everything · common words = 1 token · rare words split intelligently · no OOV.

The dominant choice — used by every modern LLM.

The trade-off subwords nail. Frequent words get their own token (fast lookup). Rare words decompose into pieces. Unknown words still tokenize via the byte-level fallback. No vocabulary blow-up.
Part 4 · 1/4

Three subword tokenizer families

FamilyAlgorithmUsed byVocab size
BPE · Byte-Pair EncodingIteratively merge most-frequent pair of symbols. Byte-level variant covers any Unicode.GPT-2/3/4 · GPT-4o · Llama · Mistral50K – 200K
WordPieceLike BPE but merges by likelihood gain, not raw frequency. Marks subword pieces with ##.BERT · DistilBERT · DeBERTa~30K
SentencePiece · UnigramProbabilistic — keeps a probability per token, prunes vocabulary to maximize likelihood. Treats whitespace as a regular character.T5 · FLAN-T5 · Gemini · multilingual mT532K – 256K

BPE in one paragraph

Start with characters → repeatedly merge the most frequent adjacent pair → save the merge rule. After N merges you have a vocabulary of frequent subwords.

Why it matters in practice

The same prompt costs different token counts on different models. Always count tokens with the model's own tokenizer (e.g., tiktoken for OpenAI) before estimating cost.

Part 4 · 3/4

Token budget management — practical

import tiktoken

enc = tiktoken.encoding_for_model("gpt-4o-mini")  # o200k_base

def count_tokens(text: str) -> int:
    return len(enc.encode(text))

system_prompt = "You are a banking assistant..."
user_query = "Summarize this 50-page policy document..."
documents = open("policy.txt").read()

budget = 128_000   # context window
reserved_for_output = 4_000

input_tokens = (
    count_tokens(system_prompt) +
    count_tokens(user_query) +
    count_tokens(documents)
)

available = budget - reserved_for_output
if input_tokens > available:
    # truncate, summarize, or chunk
    documents = enc.decode(enc.encode(documents)[:available - 2000])
Lab 1.4 §4 + §5 covers this end-to-end with tiktoken and HuggingFace tokenizers, including safe truncation for BERT-class models with max_length=512.
Rule of thumb. Reserve ~10% of context for output. Reasoning models need much more — sometimes 50%+ of max_tokens for hidden thinking.

Part 5

Generation &
sampling

How a decoder-only model turns a prompt into output, and what every sampling parameter actually does.

Autoregressive generation — step by step

Causal masking — each token only sees tokens to its left. That's why decoder-only models can stream tokens one-at-a-time as they're generated.
Part 5 · 1/4

Sampling — turning probabilities into a token

Greedy

Always pick argmax.

Deterministic. Risk: gets stuck in repetition.

Temperature

Divide logits by T before softmax.

T < 1 = sharper · T > 1 = flatter / more creative.

Top-k

Keep only top-k tokens, sample.

Hard cap. Rarely tuned in practice.

Top-p (nucleus)

Keep smallest set of tokens whose cumulative prob ≥ p.

Adaptive — cuts the long tail without hard caps.

Part 5 · 3/4

Stopping & control

Three ways generation ends

  • EOS token sampled — model decides it's done
  • max_tokens hit — hard cap reached
  • stop_sequences matched — your custom delimiter found

Reasoning-model variant

  • Hidden thinking ends when the model emits an "answer-start" token
  • Then user-facing answer streams normally
  • max_tokens must cover BOTH thinking + answer
response = client.responses.create(
    model="gpt-5-mini",
    input=user_prompt,
    reasoning={"effort": "high"},
    max_output_tokens=4000,        # ← needs headroom for hidden thinking
    stop=["END_OF_REPORT"],        # ← optional custom stop
    seed=42,                       # ← reproducibility for eval runs
)
# response.usage shows breakdown:
#  - input_tokens
#  - output_tokens         (visible)
#  - reasoning_tokens      (hidden but charged)
Reasoning tokens are billed but not returned. A reasoning_effort=high call can produce 5K–20K hidden tokens — sometimes 10× the visible output. Budget accordingly.

Part 6 · 1/2

Long context isn't free

The quadratic cost

Self-attention computes an n × n attention matrix. Doubling context = 4× compute & memory. Even with optimizations (FlashAttention, ring attention), the dollar cost scales linearly per token in the API.

Cost example

A 1M-token Gemini 2.5 Pro call at $1.25/M input ≈ $1.25 per call. A 2K Sonnet call ≈ $0.006. 200× ratio — same model family.

When NOT to use long context

  • RAG is cheaper. Retrieve only the relevant 5–10 chunks instead of stuffing everything.
  • Lost-in-the-middle. Models often miss content in the middle of very long contexts.
  • Latency. Long input = long time-to-first-token, even with prompt caching.
  • Iteration cost. Each prompt edit re-bills the long input — cache it or shorten it.
Long context techniques (under the hood). Modern long-context models use FlashAttention (memory-efficient attention), Rotary Position Embeddings (extrapolation beyond training length), and sometimes sliding-window or sparse attention. These don't change your API surface — they change what's economically possible.
Rule: If RAG can answer the question with 5K tokens, don't use a 1M-token model with the whole corpus stuffed in. Long context is a tool, not a default.

Part 7

Scaling laws
& emergence

Why bigger models keep getting better — and the surprising abilities that show up only at scale.

Scaling laws — Kaplan & Chinchilla

Kaplan et al. (2020)

Test loss decreases as a power law in three variables: parameters, dataset size, and compute. Scale them and quality improves predictably.

Chinchilla finding (2022)

Most models in 2020–21 were over-parameterized for the data they were trained on. Optimal ratio: ~20 tokens per parameter. Smaller-but-trained-longer models often beat bigger ones.

Modern frontier models (Llama 3.3, Claude 4.x) train on far more tokens than Chinchilla-optimal — sacrificing efficiency for downstream quality.
Part 7 · 1/2

Emergent abilities

Some abilities appear suddenly at certain scales — not gradually. Below the threshold, near-random performance; above it, sharp jump.

Examples

  • Multi-step arithmetic — ~10B+ params
  • In-context learning — ~10B+ params (GPT-3 moment)
  • Instruction following — needs RLHF + scale
  • Chain-of-thought reasoning — emerges with scale, then taken further by reasoning training
Caveat: some "emergence" turns out to be a measurement artifact (exact-match metrics making smooth gains look sudden). Whether it's truly discontinuous is debated — but the practical jumps are real.
Practical impact. "It worked great on Llama 70B, why not on Llama 8B?" — sometimes the answer is: that capability lives above a scale threshold the smaller model is below.

Part 8

API pricing
& cost optimization

What you actually pay for, where the savings hide, and how to budget for a 50K-msg/day banking pipeline.

Pricing model — the basics

What you pay for

  • Input tokens — system + user + retrieved docs
  • Output tokens — what the model generates
  • Reasoning tokens (reasoning models only) — hidden chain-of-thought, billed but not returned
  • Cached input tokens — re-used system prompts at 10–25% of normal price
  • Image / audio inputs — counted as token equivalents

Why output tokens cost more

Generation is sequential — one token at a time, each requiring a full forward pass. Input is processed in parallel.

Output ≈ 3–5× the price of input across most providers.

# Cost estimation (per call)
input_cost  = (prompt_tokens     / 1_000_000) * price_in_per_M
output_cost = (completion_tokens / 1_000_000) * price_out_per_M

# Reasoning models add hidden tokens
if model.startswith("gpt-5") or "thinking" in model:
    reasoning_cost = (reasoning_tokens / 1_000_000) * price_out_per_M
    total = input_cost + output_cost + reasoning_cost
else:
    total = input_cost + output_cost

Part 8 · 2/3

Cost optimization patterns

① Prompt caching

Anthropic, OpenAI: long static system prompts can be cached → cached tokens billed at ~10–25% of normal input price.

Wins big for chat with a long persona, RAG with a fixed instruction template, and tool-calling with a long tool definition.

② Batch API

OpenAI & Anthropic: submit a batch of requests, get results within 24h at ~50% discount.

Use for: nightly evaluation runs, dataset labeling, offline summarization.

③ Cascade routing

Cheap fast model handles the easy 80% — escalate to frontier only when confidence is low or the task is complex.

Often cuts cost 5–10× while maintaining quality.

④ Right-size the model

Don't use Sonnet where Haiku works. Don't use a reasoning model for classification.

Lab 1.3 §8 has the back-of-envelope math for a 50K-msg/day intake bot.

Order to apply. 1) Right-size first — biggest lever. 2) Add cascade routing if quality varies. 3) Add caching when prompts have stable prefixes. 4) Batch API for offline work.
