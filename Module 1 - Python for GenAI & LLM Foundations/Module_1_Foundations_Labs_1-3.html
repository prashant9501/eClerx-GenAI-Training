Part 1 · 2/3

Why now? Three converging forces

① Architecture

The transformer (2017) replaced sequential recurrence with parallel attention — finally fits GPUs at scale.

Without it, training a 100B model is impossible.

② Scale

Scaling laws: more params + more data + more compute → predictably better models.

GPT-3 (175B, 2020) showed the curve had not flattened.

③ Alignment

RLHF + instruction tuning (2022) turned a "next-word predictor" into a tool that follows human intent.

This is the ChatGPT moment.

Part 2 · 1/5

The "ChatGPT moment" — what actually changed

GPT-3 was a raw next-word predictor. Asking it "what is RAG?" might give you the question continued, not answered. Two training stages turned it into ChatGPT.

Modern alternatives. DPO (Direct Preference Optimization) and RLAIF (AI feedback) replace PPO in many newer models — same idea, less compute.

Part 3 · 1/4

How modern T2I works — intuition

Diffusion in one paragraph

Start from pure noise. The model has been trained to iteratively de-noise, conditioned on a text embedding. Each step gets the image slightly closer to "what the prompt describes." After ~25–50 steps you have a plausible image.

Modern T2I is almost always latent diffusion — diffusion runs in a compressed latent space (cheaper, faster), with a VAE bridging to pixel space.

Multimodal LLMs — when text models grew eyes

Modern frontier LLMs accept images as input: a single API call takes {image + text}, returns text.

# Claude Vision (covered in Lab 6.1)
response = client.messages.create(
    model="claude-sonnet-4-5",
    messages=[{
        "role": "user",
        "content": [
            {"type": "image", "source": {"type": "base64",
                                         "media_type": "image/png",
                                         "data": image_b64}},
            {"type": "text", "text": "Extract every line item from this invoice as JSON."}
        ]
    }]
)

Practical use cases

Document OCR · invoice extraction · chart analysis · diagram reading · UI testing · content moderation · slide-deck Q&A.

Trade-off

Vastly more flexible than dedicated OCR (Tesseract, Textract) — but more expensive per call. Use vision LLMs for variable-layout docs; specialized OCR for high-volume fixed-layout.

Part 4

Prompt engineering

A pragmatic primer. Module 2 (Day 2) goes deep — these slides give you the conceptual frame.

What prompt engineering is — and isn't

✓   What it IS

  • Writing instructions LLMs reliably follow
  • Choosing the right system / user split
  • Adding few-shot examples when the format is unusual
  • Asking for structured output (JSON, XML, Pydantic)
  • Iterating: read failures → adjust → retest
  • A repeatable engineering process, not folklore

✗   What it ISN'T

  • "Magic words" copied from Reddit
  • A substitute for retrieval when the model lacks knowledge
  • A substitute for fine-tuning for new behaviors at scale
  • A one-and-done — prompts drift as models update
  • Just for chat — also crucial for tool calls, agents, evaluation
A prompt is just code. Treat it like code: version it, test it, monitor it.
Part 4 · 1/4

Three prompting paradigms

Zero-shot

Just describe the task. Works on capable models for common tasks.

Classify the sentiment
(positive · negative · neutral):

"The food was cold."

Few-shot

Show 2–8 examples of the input → output pattern. Helps with niche formats.

"$45.99 → ₹4,180"
"$120.00 → ₹11,000"
"$8.50 → ?"

Chain-of-Thought

"Let's think step by step." Lets the model verbalize reasoning before answering.

"If 3 trains depart at...
Think step by step.
Show your work,
then give final answer."
On reasoning models (gpt-5-mini, o1) CoT happens internally — you don't need "let's think step by step." On non-reasoning models it still helps for math & multi-step logic.

Part 5

RAG — Retrieval-
Augmented Generation

When the model doesn't know enough — give it a reference card. The default 80% solution for enterprise GenAI.

The knowledge problem

An LLM only "knows" what was in its training data. So it fails on:

Recent events

Anything after the training cutoff date.

Private data

Your bank's policies, customers, codebase.

Long-tail facts

Obscure entities, niche regulations, internal jargon.

Exact extraction

Verbatim quotes, citations, legal clauses.

Option A · Stuff into prompt

Cheap, simple, doesn't scale beyond a few docs.

Option B · RAG ★

Retrieve relevant chunks at call time. The default 80% solution.

Option C · Fine-tune

Adapt model weights. Powerful, expensive, last resort.

Part 6

Fine-tuning
when, why, how

Four flavors, one decision rule. Most enterprise teams don't need this — but you should know when they do.

The four flavors of fine-tuning

MethodWhat changesCostWhen to use
Full fine-tune (SFT)All parametersVery highAlmost never — only mega-orgs with massive datasets
LoRA / QLoRATiny adapter matrices (~1% of params)Low (single GPU)Domain adaptation · style transfer · task-specific
Instruction tuningAdapter or fullMediumNew behavior class (e.g., always answer in legal-memo format)
RLHF · DPOAdapter or fullHigh (data + compute)Aligning preferences (helpfulness · safety · brand voice)
Reality check. For 80% of enterprise problems, strong prompt + RAG > fine-tune. Reach for fine-tuning only when prompt+RAG plateaus, or you need lower latency / cost at extreme scale.
Part 6 · 1/2

When does fine-tuning earn its keep?

Good reasons to fine-tune

  • Latency at scale. Smaller fine-tuned model replaces a frontier API call.
  • Cost at scale. Same — small model + LoRA can hit 90% of frontier quality at 1% of cost.
  • Highly specific format. Always emit a 17-section legal-memo structure.
  • Tone & brand voice. Hard to capture in a prompt at scale.
  • On-prem requirement. Open weights on your GPUs.

Bad reasons (try prompt+RAG first)

  • "It hallucinates." → That's a RAG problem.
  • "It doesn't know our policies." → RAG.
  • "It's not creative enough." → Tune temperature / change models.
  • "It got the format wrong." → Pydantic + structured output.
  • "It's slow." → Switch to a smaller / faster model first.
Premature fine-tuning is the root of most "GenAI doesn't work" stories at companies that didn't first squeeze prompt + RAG.

Part 7 · 1/5

Why use multiple providers?

① Reliability

Provider outages happen weekly. Fallback chains keep your service alive.

② Cost

3–10× price differences for similar quality. Use cheap fast model first, escalate only when needed.

③ Latency

Groq's LPU is 5–10× faster than OpenAI for Llama. Use it for in-loop calls.

④ Compliance

Some banks cannot send data to OpenAI Cloud. Bedrock keeps the call inside their AWS VPC.

Lab 1.2 builds an OpenAI → Groq fallback. Lab 1.3 shows multi-provider structured-output fallback. Both labs are practical instances of the four reasons above.

Structured outputs — production reliability

TierReliabilityMechanism
1 · Native schemaHighestProvider validates against your Pydantic schema
2 · JSON modeHighProvider guarantees valid JSON; you describe the shape in prompt
3 · Prompt-onlyLowestTrust the model to follow "respond as JSON…"
from pydantic import BaseModel

class CustomerIntent(BaseModel):
    category: str
    urgency: int   # 1-5
    sentiment: str
    summary: str

result = openai_client.responses.parse(
    model="gpt-5-mini",
    input=[{"role": "user",
            "content": user_msg}],
    text_format=CustomerIntent,
)

result.output_parsed.category   # typed
result.output_parsed.urgency    # int
Why this matters. Once you adopt Tier 1, you stop writing try: json.loads(...) except: …. Your downstream code consumes typed Pydantic objects, not strings.
